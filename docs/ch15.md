## 1. 概述 (Overview)

本章聚焦於 vibe coding 的「中層迴圈」——那些跨越數小時到數天的開發會話轉換 (session transitions)。不同於 inner loop 的即時編碼互動，middle loop 處理的是 AI 助手在會話之間的「記憶喪失」問題：如何在 AI 每次重新開始時，確保專案狀態、架構決策與品質標準能夠延續。本章以 **Prevent-Detect-Correct** 三層架構，系統性地解決多會話協作中的 context 管理、code review、多 agent 協調、以及工作流程自動化等核心挑戰。

對於已經在使用 Claude Code、GitHub Copilot 或 Gemini 的 RD 來說，這一章是從「一個人用一個 AI」升級到「一個人指揮多個 AI agent」的關鍵轉折點——你將從 developer 轉變為 tech lead。

### 框架二：Memento Method — Session Transition Protocol

**核心概念**：像電影 Memento 的主角一樣，你必須在每次會話結束前，將所有關鍵資訊「刺青」到外部文件中。

```
  Session N                    Transition                   Session N+1
+-------------+           +------------------+           +-------------+
|             |           |                  |           |             |
| Active Work |---(1)---> | Context < 50%?   |---(5)---> | Fresh Start |
|             |           |   YES: Trigger   |           |             |
| - Code edits|           |   externalization|           | - Load specs|
| - Test runs |           |                  |           | - Load rules|
| - Decisions |---(2)---> | Write spec.md:   |---(6)---> | - Resume    |
|             |           |  - Current state |           |   from plan |
|             |           |  - Next steps    |           |             |
|             |---(3)---> |  - Decisions made|---(7)---> | Verify      |
|             |           |  - Blockers      |           | continuity  |
|             |           |  - Test status   |           |             |
|             |---(4)---> | Update AGENTS.md |           |             |
|             |           | if new rules     |           |             |
+-------------+           +------------------+           +-------------+
```

**何時使用**：每次 context window 接近 50% 使用量時；每次結束工作會話前；每次切換到不同任務前。

**TypeScript 範例 -- Session State Exporter**：

```typescript
// session-state.ts -- 自動化 session transition 的輔助工具
interface SessionState {
  timestamp: string;
  taskDescription: string;
  completedSteps: string[];
  pendingSteps: string[];
  decisionsLog: DecisionEntry[];
  blockers: string[];
  testStatus: TestSummary;
  branchInfo: BranchInfo;
}

interface DecisionEntry {
  decision: string;
  reasoning: string;
  alternatives: string[];
}

interface TestSummary {
  passed: number;
  failed: number;
  skipped: number;
  failingTests: string[];
}

interface BranchInfo {
  current: string;
  base: string;
  uncommittedFiles: string[];
}

function generateSessionMarkdown(state: SessionState): string {
  return `# Session State -- ${state.timestamp}

## Current Task
${state.taskDescription}

## Completed
${state.completedSteps.map(s => `- [x] ${s}`).join('\n')}

## Pending
${state.pendingSteps.map(s => `- [ ] ${s}`).join('\n')}

## Key Decisions
${state.decisionsLog.map(d =>
  `### ${d.decision}\n- **Why**: ${d.reasoning}\n- **Alternatives considered**: ${d.alternatives.join(', ')}`
).join('\n\n')}

## Blockers
${state.blockers.map(b => `- WARNING: ${b}`).join('\n')}

## Test Status
- Passed: ${state.testStatus.passed}
- Failed: ${state.testStatus.failed}
- Skipped: ${state.testStatus.skipped}
${state.testStatus.failingTests.length > 0
  ? `\n### Failing Tests\n${state.testStatus.failingTests.map(t => `- ${t}`).join('\n')}`
  : ''}

## Branch Info
- Current: \`${state.branchInfo.current}\`
- Base: \`${state.branchInfo.base}\`
- Uncommitted: ${state.branchInfo.uncommittedFiles.length} files
`;
}

// 使用範例：在 Claude Code 會話結束前執行
// Prompt: "Please export our session state to spec/session-2025-06-15.md"
```

### 框架三：Multi-Agent Coordination Matrix

```
                    Shared Resources
                    Low          High
                +----------+----------+
      Low       | PARADISE | CAUTION  |
  Task          | Run free | Sequence |
  Overlap       | parallel | critical |
                |          | sections |
                +----------+----------+
      High      | CAREFUL  | DANGER   |
                | Monitor  | Partition|
                | merges   | or       |
                |          | serialize|
                +----------+----------+

  Legend:
  PARADISE  = Independent agents, independent resources -> max parallel
  CAUTION   = Independent tasks but shared files/ports -> need mutex
  CAREFUL   = Overlapping tasks but separate files -> merge strategy needed
  DANGER    = Overlapping tasks AND shared resources -> must partition first
```

**Python 範例 -- Agent Coordination Tracker**：

```python
# agent_coordinator.py -- 多 agent 狀態追蹤系統
from dataclasses import dataclass, field
from datetime import datetime
from enum import Enum
from typing import Optional

class AgentStatus(Enum):
    IDLE = "idle"
    WORKING = "working"
    WAITING_INPUT = "waiting_input"
    REVIEWING = "reviewing"
    ERROR = "error"

@dataclass
class AgentState:
    name: str
    role: str  # e.g., "bugs", "typescript-client", "emacs"
    branch: str
    workspace_dir: str
    status: AgentStatus = AgentStatus.IDLE
    current_task: str = ""
    work_queue: list[str] = field(default_factory=list)
    files_touched: set[str] = field(default_factory=set)
    last_updated: Optional[datetime] = None

class AgentCoordinator:
    """Steve 風格的 central command post: 追蹤所有 agent 狀態"""

    def __init__(self):
        self.agents: dict[str, AgentState] = {}

    def register_agent(self, agent: AgentState) -> None:
        self.agents[agent.name] = agent
        agent.last_updated = datetime.now()

    def detect_contention(self) -> list[dict]:
        """偵測 agent 間的資源衝突"""
        conflicts = []
        agents = list(self.agents.values())

        for i, a1 in enumerate(agents):
            for a2 in agents[i + 1:]:
                # 檢查 file contention
                shared_files = a1.files_touched & a2.files_touched
                if shared_files:
                    conflicts.append({
                        "type": "file_contention",
                        "agents": [a1.name, a2.name],
                        "resources": list(shared_files),
                        "severity": "HIGH" if len(shared_files) > 3
                                    else "MEDIUM"
                    })

                # 檢查 branch confusion
                if a1.branch == a2.branch and a1.branch != "main":
                    conflicts.append({
                        "type": "branch_collision",
                        "agents": [a1.name, a2.name],
                        "branch": a1.branch,
                        "severity": "CRITICAL"
                    })

                # 檢查 workspace overlap
                if a1.workspace_dir == a2.workspace_dir:
                    conflicts.append({
                        "type": "workspace_overlap",
                        "agents": [a1.name, a2.name],
                        "directory": a1.workspace_dir,
                        "severity": "HIGH"
                    })

        return conflicts

    def get_idle_agents(self) -> list[AgentState]:
        """找出閒置中的 agent，可以指派 self-review 任務"""
        return [a for a in self.agents.values()
                if a.status == AgentStatus.IDLE]

    def assign_self_review(self, agent_name: str) -> list[str]:
        """
        指派 self-review 任務給閒置 agent
        (書中的 'keep agents busy' 策略)
        """
        return [
            "Run all tests again and report any failures",
            "Review code for missing edge cases",
            "Improve existing test coverage",
            "Check error handling and robustness",
            "Clean up temp files, debug logs, and untracked files",
            "Write a markdown summary of current progress",
        ]

    def export_dashboard(self) -> str:
        """輸出 command post dashboard"""
        lines = ["# Agent Command Post",
                 f"Updated: {datetime.now()}", ""]

        for name, agent in self.agents.items():
            lines.append(f"## {agent.name} ({agent.role})")
            lines.append(f"- Status: **{agent.status.value}**")
            lines.append(f"- Branch: `{agent.branch}`")
            lines.append(f"- Task: {agent.current_task}")
            lines.append(f"- Queue: {len(agent.work_queue)} items")
            lines.append("")

        conflicts = self.detect_contention()
        if conflicts:
            lines.append("## ALERTS")
            for c in conflicts:
                lines.append(
                    f"- [{c['severity']}] {c['type']}: "
                    f"{' vs '.join(c['agents'])}"
                )
            lines.append("")

        return "\n".join(lines)
```

**TypeScript 範例 -- Tracer Bullet Pattern**：

```typescript
// tracer-bullet-example.ts
// 情境：驗證 AI 能否處理 shell command integration

import { execFileSync } from 'child_process';

// Step 1: 最小化 tracer bullet -- 只做一件事
function tracerBullet_shellList(): boolean {
  try {
    // Gene 的案例：最簡單的 fork + pipe + display
    const result = execFileSync('ls', ['-a'], { encoding: 'utf-8' });
    console.log('Tracer bullet SUCCESS:', result.substring(0, 100));
    return true;
  } catch (error) {
    console.error('Tracer bullet FAILED:', error);
    return false;
  }
}

// Step 2: 如果 tracer 成功，才擴展功能
async function expandFromTracer(): Promise<void> {
  if (!tracerBullet_shellList()) {
    console.log('AI cannot handle shell integration.');
    console.log('Switching to manual implementation...');
    return;
  }

  // Tracer 成功 -> 繼續讓 AI 擴展
  // "Now add the LLM draft generation command..."
  console.log('Tracer bullet passed! Expanding to full implementation...');
}

// Step 3: 如果 tracer 失敗，記錄原因供未來參考
interface TracerResult {
  domain: string;
  timestamp: Date;
  success: boolean;
  failureReason?: string;
  recommendation:
    | 'expand_with_ai'
    | 'manual_implementation'
    | 'try_different_approach';
}
```

## 5. 最難理解的三個部分 (3 Hardest Sections)

### 難點 1：Context Saturation 與 Proactive Externalization 的時機判斷

- **為什麼難**：大多數開發者習慣讓工具自動管理記憶。你不會主動告訴 IDE「請記住我剛才的重構決策」。但 AI coding agent 的 context window 是有限且衰退的——Steve 的團隊發現 50% 填充率時就開始遺忘關鍵指令。判斷「什麼時候該停下來 externalize」是一個需要經驗累積的直覺。

- **白話解釋**：想像你在用白板和一個會議室的人討論架構設計。白板空間有限，而且——這裡是關鍵——當白板寫到一半滿的時候，這個人會開始「選擇性失明」，看不到白板上方的內容。你不能等白板全滿才拍照存檔，你必須在白板半滿時就停下來，把重要的東西抄到筆記本上，然後擦掉白板重新開始。那個「筆記本」就是你的 spec file。

- **程式碼示範**：

```typescript
// context-monitor.ts -- 模擬 context window 管理策略

interface ContextWindow {
  maxTokens: number;
  usedTokens: number;
  criticalInstructions: string[];
}

function shouldExternalize(ctx: ContextWindow): {
  shouldStop: boolean;
  urgency: 'low' | 'medium' | 'high' | 'critical';
  action: string;
} {
  const usagePercent = ctx.usedTokens / ctx.maxTokens;

  if (usagePercent >= 0.8) {
    return {
      shouldStop: true,
      urgency: 'critical',
      action: 'STOP NOW. Write spec file immediately. '
            + 'Auto-compaction may trigger and lose critical context.'
    };
  }

  if (usagePercent >= 0.5) {
    return {
      shouldStop: true,
      urgency: 'high',
      action: 'Context degradation likely. Externalize state: '
            + '1) Current plan 2) Decisions made '
            + '3) Test status 4) Next steps'
    };
  }

  if (usagePercent >= 0.3) {
    return {
      shouldStop: false,
      urgency: 'medium',
      action: 'Consider wrapping up current sub-task. '
            + 'Plan your next externalization point.'
    };
  }

  return {
    shouldStop: false,
    urgency: 'low',
    action: 'Continue working. Monitor context usage.'
  };
}

// 實務提示：在 Claude Code 中，你可以觀察進度條
// 或詢問 "How much context have we used?"
// 當看到 "Context window is getting large" 訊息時
// 立即觸發 externalization
```

### 難點 2：多 Agent 從 2 到 4 的非線性複雜度跳躍

- **為什麼難**：Steve 發現從 2 agent 到 4 agent 不是「2 倍複雜」，而是「10 倍以上的組織工作」。這違反了大多數人的線性直覺。如果 2 個 agent 的認知負擔是 X，我們直覺會認為 4 個是 2X，但實際上是 10X+。這是因為 agent 間的潛在交互作用呈組合爆炸增長。

- **白話解釋**：想像你是一個指揮交通的警察。管理一個十字路口（2 個方向）很直觀——紅綠燈就行了。但管理一個五岔路口（4-5 個方向），你需要的不只是更多紅綠燈——你需要全新的交通規則、圓環設計、標誌系統、甚至可能需要一個交通控制中心。這就是 Steve 必須建構 "central command post" 的原因。

- **程式碼示範**：

```python
# complexity_calculator.py -- 展示多 agent 複雜度的組合爆炸
from math import comb

def calculate_agent_complexity(n_agents: int) -> dict:
    """
    計算 n 個 agent 間的潛在交互複雜度

    Pair interactions: C(n, 2) = n*(n-1)/2
    即使只看 pair-wise 衝突，複雜度也是 O(n^2)
    但實際上還有三方衝突、四方衝突等...
    """
    pair_interactions = comb(n_agents, 2)
    triple_interactions = comb(n_agents, 3)

    # 每個 interaction 需要的管理工作估算
    management_overhead_per_pair = 4  # 四種檢查
    total_overhead = pair_interactions * management_overhead_per_pair

    return {
        "agents": n_agents,
        "pair_interactions": pair_interactions,
        "triple_interactions": triple_interactions,
        "estimated_management_tasks": total_overhead,
        "relative_to_2_agents": (
            total_overhead / (1 * management_overhead_per_pair)
        )
    }

# 展示非線性增長
for n in [2, 3, 4, 5, 8]:
    result = calculate_agent_complexity(n)
    print(
        f"{n} agents: {result['pair_interactions']} pairs, "
        f"{result['estimated_management_tasks']} mgmt tasks, "
        f"{result['relative_to_2_agents']:.1f}x vs 2-agent baseline"
    )

# Output:
# 2 agents:  1 pairs,   4 mgmt tasks,  1.0x vs 2-agent baseline
# 3 agents:  3 pairs,  12 mgmt tasks,  3.0x vs 2-agent baseline
# 4 agents:  6 pairs,  24 mgmt tasks,  6.0x vs 2-agent baseline
# 5 agents: 10 pairs,  40 mgmt tasks, 10.0x vs 2-agent baseline
# 8 agents: 28 pairs, 112 mgmt tasks, 28.0x vs 2-agent baseline

# Steve 說 "over 10x" -- 完全符合組合數學的預測！
# 教訓：不要從 2 直接跳到 4，先穩定 3 agent 的工作流再擴展
```

### 難點 3：如何判斷 Code Base 正在走向 "Eldritch Horror" 狀態

- **為什麼難**：Gene 的 writer's workbench 從「看起來運作正常」到「3000 行 monolithic function、三天重寫」之間，沒有明顯的分界線。問題在於 AI 生成的 code 往往「能跑」，但每次新增功能都在悄悄侵蝕模組邊界。就像溫水煮青蛙——等你發現的時候已經太晚了。

- **白話解釋**：你家的電線系統一開始是規規矩矩的。然後你請了一個超快的水電工（AI），每次你說「幫我加一個插座」，他不但加了插座，還把線路接到冰箱的電路上，再串連到浴室的燈光。每個單獨的改動都「能用」，但三個月後你拔掉客廳的檯燈插頭，廚房的冰箱就停了。這就是 AI 生成的 code 如何破壞模組邊界。

- **程式碼示範**：

```typescript
// code-health-detector.ts -- 偵測 code base 走向 horror 的指標

interface CodeHealthMetrics {
  maxFunctionLength: number;     // 最長函式的行數
  avgFunctionLength: number;     // 平均函式行數
  maxFileLength: number;         // 最長檔案的行數
  crossModuleImports: number;    // 跨模組的 import 數量
  circularDependencies: number;  // 循環依賴數量
  testCoverage: number;          // 測試覆蓋率 (0-1)
  moduleBoundaryViolations: number;
}

function diagnoseCodeHealth(metrics: CodeHealthMetrics): {
  status: 'healthy' | 'warning' | 'critical' | 'eldritch_horror';
  recommendations: string[];
} {
  const recommendations: string[] = [];
  let score = 0;

  // Gene 的案例: 3000 行的函式 -> 絕對是 horror 等級
  if (metrics.maxFunctionLength > 500) {
    score += 3;
    recommendations.push(
      `CRITICAL: Largest function is ${metrics.maxFunctionLength} lines. `
      + `Split into modules IMMEDIATELY. `
      + `Gene's 3000-line horror took 3 days to fix.`
    );
  } else if (metrics.maxFunctionLength > 200) {
    score += 1;
    recommendations.push(
      `WARNING: Functions exceeding 200 lines. `
      + `AI struggles to read these in full. `
      + `Refactor before they grow further.`
    );
  }

  // 檔案大小: AI 的 context window 限制
  if (metrics.maxFileLength > 2500) {
    score += 2;
    recommendations.push(
      `File too large for AI (>${metrics.maxFileLength} lines). `
      + `AI will resort to grepping, losing holistic understanding.`
    );
  }

  // 模組邊界違反 -- 最危險的指標
  if (metrics.moduleBoundaryViolations > 5) {
    score += 3;
    recommendations.push(
      `${metrics.moduleBoundaryViolations} module boundary violations. `
      + `Modules are FUSING. This destroys the "O" in FAAFO.`
    );
  }

  // 測試覆蓋率
  if (metrics.testCoverage < 0.3) {
    score += 2;
    recommendations.push(
      `Test coverage at ${(metrics.testCoverage * 100).toFixed(0)}%. `
      + `Without tests, you cannot detect when AI breaks things. `
      + `Gene learned this the hard way -- add TDD NOW.`
    );
  }

  const status = score >= 6 ? 'eldritch_horror'
               : score >= 4 ? 'critical'
               : score >= 2 ? 'warning'
               : 'healthy';

  return { status, recommendations };
}

// 使用方式: 每天跑一次，或在每次 AI session 結束後跑
// 搭配 Git hooks 或 CI/CD pipeline 自動化執行
```

## 6. 本章小結與架構啟示

### 從 Developer 到 Tech Lead 的角色轉變

```
  BEFORE (Traditional Dev)        AFTER (Vibe Coding Middle Loop)
  +-------------------+           +---------------------------+
  | You write code    |           | You DIRECT agents to      |
  | You debug         |           |   write code              |
  | You test          |           | You COORDINATE multiple   |
  | You review        |           |   parallel workstreams    |
  | You deploy        |           | You DESIGN agent-friendly |
  |                   |           |   architecture            |
  | Role: IC Dev      |           | You BUILD automation      |
  |                   |           |   infrastructure          |
  |                   |           |                           |
  |                   |           | Role: Tech Lead / Director|
  +-------------------+           +---------------------------+
```

### 五大架構啟示

1. **Memory Infrastructure is Architecture**：AGENTS.md、spec files、session state exports 不是「nice to have」——它們是你的系統架構的一部分。就像 database schema 定義了資料如何被存取，memory infrastructure 定義了 AI 如何理解你的專案。

2. **Design for AI Grain, Not Against It**：選擇 popular languages/frameworks、維持 conventional project structures、保持函式和檔案在 AI 可讀範圍內。這不是妥協，這是「Design for Manufacturing」——就像汽車工程師為裝配線工人設計零件。

3. **Contention is Combinatorial**：多 agent 的管理複雜度是 O(n^2) 而非 O(n)。在從 2 agent 跳到 4 agent 之前，先建立 coordination infrastructure（command post、naming conventions、merge protocols）。

4. **Idle Time is Review Time**：Agent 閒置時不要讓它等待——指派 self-review、test improvement、edge case hunting。Anthropic 的研究顯示 2-3 輪迭代能顯著提升品質。這是免費的品質提升。

5. **Optionality Demands Modularity**：NK/t 公式告訴我們，option value 來自模組化 (N)、並行實驗 (K)、和快速迭代 (t)。每一次你允許 AI 破壞模組邊界，你都在降低 N，摧毀 optionality。Gene 花了三天重建模組化——這個成本要記住。

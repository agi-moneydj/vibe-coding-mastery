# 第 6 章：四個 Vibe Coding 案例研究 (Four Case Studies in Vibe Coding)

## 1. 概述 (Overview)

本章是全書最「接地氣」的一章，透過四個真實案例展示 vibe coding 在不同情境下的實際表現：個人 side project（Luke 的 CNC firmware uploader）、回歸開發者（Christine 的 Google Calendar 匯出）、以及兩個企業級大規模導入（Adidas 700 人開發者試驗、Booking.com 3000 人團隊）。這四個案例從「一個人 + AI」到「千人團隊 + AI」，完整覆蓋了 vibe coding 的應用光譜，同時暴露了真實的痛點和成功的關鍵條件。

## 2. 學習目標 (Learning Objectives)

完成本章後，你應該能夠：

## 3. 章節大綱 (Outline)

### 3.1 案例一：Luke Burton — CNC Firmware Uploader (個人 Side Project)

- **背景**：Apple 前工程管理者，近 20 年經驗，hobby CNC 玩家
- **問題**：CNC firmware 開發環境極差
  - 測試必須在實體機器上進行（非本機）
  - 上傳 firmware 需要繁瑣的 telnet 命令
  - Unit tests 幾乎不存在
- **過程與成果**
  - 使用 Claude Code，2 小時產出 2600 行 Python
  - 包含完整文件、CLI flags、interactive mode + GNU readline
  - 花費 $50 Claude Code tokens
- **工具切換策略**
  - Claude Code 處理壓縮檔案時遇困難
  - 切換到 Cursor（同一個 Claude Sonnet 3.7 模型）
  - 餵入其他可用的 Python 程式碼作為參考，兩次嘗試後成功
- **FAAFO 體現**：Fast（2小時）、Ambitious（完整 CLI 工具）、Autonomous（一人完成）、Fun、Optionality（多工具切換）

### 3.2 案例二：Christine Hudson — 回歸 Coding (非工程師 / 回歸者)

- **背景**：2004 年機器學習碩士，15-20 年未寫程式
- **任務**：匯出 Google Calendar 到另一個 Google 帳號
- **三種嘗試路徑**
  - Christine：Google Apps Script → **成功**
  - Steve：複製 Christine 的方法 → 失敗（authentication 問題）
  - Gene：Python + Google Colab → 失敗（OAuth consent screen 卡關）
- **關鍵洞見**
  - Google Apps Script 的優勢：**已內建身份驗證**，免除 OAuth 地獄
  - 經驗豐富的開發者知道哪條路可以避開複雜度 → 但有時反而選錯路
  - 與外部服務的 authentication 是 vibe coding 最大的「barbed wire」
- **體驗評分**
  - 歡喜時刻：看到 AI 自動寫出程式碼 → 「+10」pure joy
  - 挫折時刻：mildly annoying，不比日常電腦問題排除更糟
  - Steve 的比喻：「vibe coding 有時像地獄版的 Disneyland——遊樂設施被壓縮成半秒，剩下的全是排隊」

### 3.3 案例三：Adidas — 700 人開發者試驗 (企業級)

- **背景**：年營收 90 億歐元，全球前五大電商品牌
- **Fernando Cornago**：Global VP of Digital and E-Commerce Technology
- **第一次試驗：慘敗**
  - 90% 開發者厭惡 AI coding assistant
  - 評語：「total waste of time」、「firefighting and troubleshooting」
  - 時間：2024 年初，工具和模型尚未成熟
- **第二次試驗：成功**
  - 70% 開發者生產力提升 20-30%（commits、PRs、feature delivery velocity）
  - 開發者自評效率提升 20-25%
  - **最重要指標：50% 增加「Happy Time」**
    - Happy Time = 開發者想做的事（coding、分析、設計）
    - Annoying Time = 不想做的事（會議、環境排錯、brittle tests、行政）
- **成功 vs. 失敗團隊的差異**
  - 成功團隊：loosely coupled architecture、clear API boundaries、fast feedback loops
  - 失敗團隊：tightly coupled、模糊的邊界、慢 feedback

### 3.4 案例四：Booking.com — 3000+ 開發者 (企業級)

- **背景**：全球最大線上旅遊平台之一，超過 3000 名開發者
- **Bruno Passos**：Group Product Manager, Developer Experience
- **痛點**：feature flags 文化的副作用
  - 大量從未使用的功能、disabled feature flags、legacy code
  - 開發者 90% 時間花在 frustrating toil
- **解決方案**：Sourcegraph AI code assistant + search tools
- **成果**
  - 30% coding efficiency 提升
  - Merge requests 小 70%
  - Review times 顯著減少
  - 30% 更多 merge requests
- **關鍵成功因素：Training**
  - 初期導入不均衡——有人愛用，有人看不到好處
  - 缺失的要素：開發者不知道如何給 AI 正確的 instructions 和 context
  - 解法：每個 business unit 進行培訓日、hackathons、workshops
  - 結果：原本猶豫的開發者轉變為熱情的日常 vibe coders

## 5. 最難理解的三個部分 (3 Hardest Sections)

### 難點 1：為什麼 Adidas 第一次試驗 90% 失敗，第二次卻 70% 成功？

- **為什麼難**：表面看起來只是「工具變好了」，但真正的差異遠比這深層。很多組織也在經歷第一次失敗後就放棄了，沒有機會進入第二次嘗試。

- **白話解釋**：想像你買了一台全新的工業級咖啡機給辦公室。第一天，所有人抱怨「這比即溶咖啡還麻煩」，因為沒人教他們怎麼用。90% 的人回去泡即溶。三個月後，你請了一位咖啡師來做培訓、寫了操作手冊、調整了預設參數。突然間 70% 的人愛上了它。工具一樣，但「使用方式」完全不同。

  Adidas 的關鍵轉變不只是模型從 GPT-3.5 時代進步到 Claude Sonnet，更重要的是：
  1. 團隊學會了如何給 AI 有效的 context
  2. 建立了衡量標準（不再靠感覺）
  3. 識別出架構對 AI 生產力的決定性影響

- **程式碼示範**：loosely coupled vs. tightly coupled 架構如何影響 AI 的有效性

  ```typescript
  // === Loosely Coupled (AI 友好 - Adidas 成功團隊) ===
  // 清晰的 API boundary，AI 可以獨立修改每個 service

  // payment-service/src/processPayment.ts
  export async function processPayment(order: Order): Promise<PaymentResult> {
    const validated = validateOrder(order);
    const charged = await chargeProvider(validated);
    return mapToResult(charged);
  }
  // AI 可以安全修改此函式，不會影響其他 service

  // === Tightly Coupled (AI 不友好 - Adidas 失敗團隊) ===
  // 一切交織在一起，AI 改一處就破壞十處

  // monolith/src/handlers/orderHandler.ts
  export function handleOrder(req: Request, res: Response) {
    const user = globalUserCache.get(req.session.id);  // 全域狀態
    const cart = user.cart;  // 深層巢狀存取
    db.query(`UPDATE orders SET status='paid' WHERE user_id=${user.id}`);  // 直接 SQL
    emailService.send(user.email, cart.items);  // 隱式依賴
    analyticsTracker.track('purchase', { ...user, ...cart }); // 更多隱式依賴
    inventorySystem.reduce(cart.items); // 又一個隱式依賴
    res.json({ ok: true });
  }
  // AI 改任何一行都可能產生連鎖反應
  ```

### 難點 2：Christine 成功而 Steve 和 Gene 失敗的真正原因

- **為什麼難**：經驗豐富的開發者反而失敗、新手反而成功，這違反直覺。很容易誤讀為「經驗不重要」，但真正的教訓更加微妙。

- **白話解釋**：這就像三個人要從 A 點到 B 點。老司機 Steve 說「走高速公路最快」，結果高速公路封了（auth 問題）。Gene 說「走省道比較穩」，結果省道在修路（OAuth consent screen）。Christine 聽了建議走小巷子（Google Apps Script），結果小巷子一路暢通到底。

  經驗豐富的人知道多條路，但有時反而選了更「專業」但更複雜的路。Christine 的成功恰恰因為她走了「最不專業但最 pragmatic」的路——**而這條路是 Steve 和 Gene 幫她選的**。經驗的真正價值不是自己選對路，而是為別人指出最簡單的路。

- **架構啟示**：

  ```
  選擇開發路徑的決策樹:

  需要存取外部服務？
  ├── 否 → 直接用 AI 寫邏輯 (最快)
  └── 是 → 該服務有已認證的內建環境嗎？
       ├── 是 → 用那個環境！(Google Apps Script, Cloudflare Workers, etc.)
       │        ↑ Christine 走的路
       └── 否 → 需要自己處理 OAuth/API Key
            ├── 熟悉該 API？ → 可能可行，但預期會卡
            │   ↑ Steve 和 Gene 走的路
            └── 不熟悉？ → 高風險，考慮換方案
  ```

### 難點 3：Booking.com 的「90% 時間花在 frustrating toil」如何被改善

- **為什麼難**：「開發者 90% 時間花在 toil」聽起來很誇張，但如果你在大型 codebase 工作過就知道這很真實。困難的是理解 AI 工具如何精確地減少這 90% 中的哪些部分。

- **白話解釋**：想像你在一個巨大的倉庫裡工作。貨架上堆滿了東西，很多標籤已經褪色（legacy code），有些架子上的東西永遠不會有人取用（disabled feature flags），走道之間雜物堆積（technical debt）。你要找一個零件，光是搜尋和理解就花了一天，實際安裝只花十分鐘。

  Sourcegraph 做的事就像給你一副「X 光眼鏡」——你可以立刻看穿整個倉庫，找到你要的東西，理解它跟什麼相連。AI agent 還能幫你清理堆積的雜物。

  **關鍵數據解讀**：
  - MR 小 70% → AI 幫開發者做更精確的改動，不再「為了安全多改一堆」
  - Review time 減少 → 小的 MR 天然更容易 review
  - 30% 更多 MR → 同樣的工作量產出更多

- **程式碼示範**：用 AI 清理 feature flag 殘留的典型場景

  ```python
  # Booking.com 式的問題：堆積的 feature flags
  # AI 可以幫忙識別和清理

  # BEFORE: 充滿殘留 feature flags 的程式碼
  def calculate_price(room, dates, user):
      base_price = room.price * len(dates)

      if feature_flags.is_enabled("exp_dynamic_pricing_v2"):  # 2年前的實驗
          base_price = dynamic_pricing_v2(room, dates)
      elif feature_flags.is_enabled("exp_dynamic_pricing_v1"):  # 3年前的實驗
          base_price = dynamic_pricing_v1(room, dates)

      if feature_flags.is_enabled("loyalty_discount_2023"):  # 已永久啟用
          discount = get_loyalty_discount(user)
          base_price *= (1 - discount)

      if feature_flags.is_enabled("covid_flexible_cancel"):  # COVID 已結束
          pass  # 空的！但沒人敢刪

      return base_price

  # AFTER: AI 輔助清理後
  def calculate_price(room, dates, user):
      base_price = dynamic_pricing_v2(room, dates)  # 確認為當前策略
      discount = get_loyalty_discount(user)           # 永久功能，移除 flag
      return base_price * (1 - discount)
  ```

### 架構啟示一：架構品質是 AI 生產力的乘數

Adidas 的數據清楚表明：loosely coupled architecture 的團隊從 AI 工具獲得的收益遠大於 tightly coupled 團隊。這不是偶然——AI 在邊界清晰、職責分明的模組中表現最好，就像人類工程師一樣。**如果你想最大化 AI 的投資回報，先投資在架構改善上**。

### 架構啟示二：培訓不是可選的，是必要的

Booking.com 最初導入不均衡，直到進行結構化培訓後才看到廣泛採用。**給開發者 AI 工具但不教他們怎麼用，等於給廚師一把新刀但不教他怎麼磨**。培訓的核心是教會兩件事：
1. 如何給 AI 提供有效的 context
2. 如何給 AI 明確的 instructions

### 架構啟示三：Happy Time 是最被低估的指標

Fernando Cornago 追蹤的「Happy Time」增加 50% 可能比任何生產力數字都重要。快樂的開發者留任率更高、創造力更強、更願意嘗試新事物。在你的組織中開始追蹤這個指標。

### 給資深開發者的直接建議

1. **從 side project 開始**——Luke 的案例告訴我們，2 小時 + $50 就能產出有價值的工具
2. **預期第一次失敗**——Adidas 的經驗告訴我們，失敗是學習的一部分，關鍵是迭代
3. **避開外部服務的 Auth 地獄**——Christine 的案例告訴我們，選對已認證的平台比硬闖 OAuth 聰明得多
4. **投資培訓**——Booking.com 告訴我們，工具 alone 不夠，人的技能提升才是關鍵
5. **衡量 Happy Time**——不只看產出量，也看開發者做有意義工作的時間比例

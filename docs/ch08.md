## 1. 概述 (Overview)

本章是 Part 2 的起手式，作者刻意將「動手做」放在理論之前。核心主張：你不再親手寫程式碼，而是以 head chef 的身分指揮 AI sous chef 完成工作。透過一系列由淺入深的練習 —— 從 bouncing ball 到 Flappy Bird —— 讓讀者親身體驗「自然語言 → 可執行程式碼」的轉換過程，建立對 vibe coding 的直覺。

對已經在用 Claude Code / Copilot 的 RD 而言，本章的價值不在練習本身，而在於它所揭示的 **mental model**：你的核心技能已從「寫語法」轉移到「有效溝通」。

## 2. 學習目標 (Learning Objectives)

完成本章後，你應該能夠：

- **辨識** chatbot（Claude / ChatGPT / Gemini）的 sandbox 執行模式與各自的能力邊界（Artifacts / Code Interpreter / Canvas）
- **理解** one-shot prompting 為何能成功（訓練資料中有大量經典實作）以及它在何時會失敗
- **實作** 從自然語言描述到可互動應用程式的完整 vibe coding 流程
- **養成** 在開發過程中隨時 "bounce off AI" 的習慣，保持 flow state

## 3. 章節大綱 (Outline)

### 3.1 你的第一次 Vibe Coding 體驗
- 使用 Claude Artifacts 生成 bouncing ball 動畫
- 漸進式迭代：改色、放大、加球、變遊戲
- 關鍵觀察：你不需要學 Canvas API、物理引擎、async model

- Flappy Bird 遊戲 —— one-shot wonder 的極限與邊界

### 3.3 何時該請 AI 幫忙
- 新任務啟動 → 與 AI 腦力激盪
- 生成程式碼後 → 請 AI 找 edge cases
- 需要測試 → 讓 AI 建立 test plan + 寫測試
- 修完 bug → 請 AI 撰寫 changelog
- 閱讀文件/程式碼 → AI 當第二雙眼睛
- 環境設定卡關 → AI walk-through

### 3.4 建議練習
- Pair vibe coding（兩人、兩 AI）
- Group vibe coding competition
- ChatGPT Voice Mode（邊遛狗邊學）

### 框架二：One-Shot Success 條件

AI 能 one-shot 成功的條件：
1. 訓練資料中有大量類似實作（經典遊戲、常見 UI pattern）
2. 需求可由單一 prompt 完整描述
3. 不需要與外部系統整合

反直覺點：AI 能從模糊 prompt 生成 WWII flight simulator，卻可能在技術上更簡單的任務卡關。

```
One-Shot 成功率光譜：

  高成功率                                    低成功率
  |                                              |
  v                                              v
  +----------+  +-----------+  +----------+  +----------+
  | 經典遊戲 |  | 標準 CRUD |  | 領域邏輯 |  | 系統整合 |
  | Flappy   |  | REST API  |  | 計算公式 |  | 多服務   |
  | Bird     |  | Todo App  |  | 財務規則 |  | 串接     |
  +----------+  +-----------+  +----------+  +----------+

  原因：訓練     原因：大量     原因：需要     原因：需要
  資料豐富       範本存在       domain        runtime
                               knowledge     context
```

### 框架三：Flow State 保持策略

```
+----------+    卡關    +------------------+
|  寫程式  | --------> |  問 AI 怎麼做     |
+----------+           +------------------+
     ^                         |
     |    繼續 flow             v
     +<-------------- AI 回答 + 程式碼
```

核心原則：**任何讓你離開 flow 的事情，都先問 AI 能不能代勞。**

### 框架四：Chatbot 平台能力比較（2025）

| 平台 | 執行環境 | 強項 | 限制 |
|------|---------|------|------|
| Claude | Artifacts | React/JS 即時渲染、多步迭代 | 無法存取外部 API |
| ChatGPT | Code Interpreter | Python 資料分析、檔案上傳 | JS 互動應用較弱 |
| Gemini | Canvas | 長 output、web search | 穩定性波動 |

選擇原則：快速 prototype 用 chatbot sandbox；需要持久化和部署用 Replit/Lovable。

## 5. 最難理解的三個部分 (3 Hardest Sections)

### 難點 1：One-Shot Wonder 的能力邊界在哪裡？

- **為什麼難**：本章展示的範例都太「神奇」了，容易讓人高估 one-shot 的泛用性。Bouncing ball、Flappy Bird 都成功，但真實工程問題往往不是這樣。
- **白話解釋**：想像 AI 是一個背了一萬道食譜的廚師。你說「做一道宮保雞丁」，他立刻就做得很好，因為他見過太多版本。但如果你說「把宮保雞丁的醬汁用在我們公司獨特的分子料理手法上」，他就可能亂搞，因為訓練資料裡沒有這個組合。
- **實務判斷**：

```typescript
// One-shot 適合的場景：
// - 經典演算法實作
// - 常見 UI component
// - 標準 CRUD API

// One-shot 不適合的場景：
// - 與你特定 codebase 的整合邏輯
// - 需要領域知識的商業規則
// - 多系統交互的 edge case 處理
```

### 難點 2：「不需要寫程式碼」不代表「不需要理解程式碼」

- **為什麼難**：本章為了降低門檻，強調「不需要 coding knowledge」。但對有經驗的 RD 來說，這可能傳遞錯誤訊號——以為可以完全不看 AI 生成的程式碼。
- **白話解釋**：你不需要自己開車，但你需要看懂地圖、知道目的地對不對。Head chef 不自己炒菜，但必須嚐味道。你需要的是「能讀懂程式碼並判斷品質」的能力，而不是「能從頭手寫」的能力。
- **架構師的觀點**：

```python
# 對有經驗的 RD，正確心態是：
# 1. 讓 AI 生成 → 快速取得 working prototype
# 2. Review 生成結果 → 判斷架構/安全性/可維護性
# 3. 迭代修正 → 用對話引導 AI 改善品質

# 不是 "fire and forget"，而是 "generate and validate"
```

### 難點 3：Sandbox 環境 vs. Production 的落差

- **為什麼難**：在 Claude Artifacts 裡跑 Flappy Bird 很簡單，但把同樣的邏輯部署到 production 是完全不同的事。本章沒有觸及這個落差。
- **白話解釋**：在自家廚房做一道菜很容易，但要開一家每天服務五百人的餐廳，你需要的不只是食譜——還有供應鏈、衛生標準、成本控制。
- **實務建議**：把本章練習當作 rapid prototyping 工具。用 sandbox 快速驗證想法，但部署前要經過 code review、testing、security audit 的完整流程。
- **程式碼示範**：

```python
# Sandbox 中可以接受的 "快樂路徑" 程式碼：
def fetch_data(url):
    response = requests.get(url)
    return response.json()

# Production 需要的品質（後續章節會展開）：
def fetch_data(url: str, timeout: int = 5) -> dict:
    """Fetch JSON data with retry, timeout, and error handling."""
    for attempt in range(3):
        try:
            response = requests.get(url, timeout=timeout)
            response.raise_for_status()
            return response.json()
        except requests.Timeout:
            logger.warning(f"Timeout on attempt {attempt + 1}")
            if attempt == 2:
                raise
        except requests.HTTPError as e:
            logger.error(f"HTTP {e.response.status_code}: {url}")
            raise
```

## 6. 本章小結與架構啟示

### 對資深 RD 的三個 Actionable Takeaways

1. **建立「AI-first」的問題解決反射**：任何任務開始前，先花 30 秒想：「我能不能用 AI 加速這件事？」大多數時候答案是肯定的。這不是偷懶，而是效率優化。

2. **Voice dictation 是被低估的加速器**：作者反覆提到用語音輸入。在實務中，這對需要描述複雜需求的場景特別有用——你可以邊思考邊說，比打字更快進入 flow。

3. **Group vibe coding session 是團隊技能提升的捷徑**：即使團隊成員已經各自使用 AI 工具，一起做 vibe coding 仍然能學到新技巧。每個人和 AI 的互動模式不同，觀察別人的做法是最快的學習方式。

### 架構師視角

本章看似簡單，但隱含一個重要的架構思維轉變：**開發流程的瓶頸從「寫程式碼」轉移到「定義問題 + 驗證結果」**。這意味著：

- Code review 的重點從「語法與邏輯」轉向「需求符合度與架構合理性」
- Testing strategy 變得比以前更重要，因為 AI 生成的程式碼需要更嚴格的驗證
- 團隊的核心能力指標從 "lines of code" 轉向 "quality of specifications"

```
傳統開發瓶頸:
  [定義問題] --> [寫程式碼 <<<] --> [測試] --> [部署]

Vibe Coding 瓶頸:
  [定義問題 <<<] --> [AI 生成] --> [驗證 <<<] --> [部署]
```

瓶頸轉移了，但並沒有消失。你省下來的「寫程式碼」時間，應該投資在「更精準的問題定義」和「更嚴格的驗證」上。

### 給已經在用 AI 工具的你

如果你已經是 Claude Code / Copilot 的日常使用者，本章最大的收穫不是技術層面的，而是 **心態校準**：

1. **你的價值不在於打字速度**：AI 讓「寫程式碼」的成本趨近於零。你的不可替代性在於「問對問題」和「判斷品質」。

2. **不要跳過練習**：即使 bouncing ball 對你來說太簡單，試試用 voice dictation 做一次。你會發現語音互動和打字互動是完全不同的 workflow，各有優勢。

3. **觀察 AI 的「額外贈送」**：作者提到 Claude 自動加了 pause/reset 按鈕。注意 AI 什麼時候會「多做」，什麼時候會「少做」——這個直覺在後續章節討論 reward function hijacking 時會非常重要。

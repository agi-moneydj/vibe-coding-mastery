              |                   |                   |

**TypeScript 範例 -- Testing Strategy Selector**：

```typescript
// testing-strategy.ts -- 根據四象限選擇測試策略

type RiskLevel = 'low' | 'medium' | 'high' | 'critical';
type TechFamiliarity = 'novice' | 'intermediate' | 'expert' | 'author';

interface TestingStrategy {
  quadrant: 'A' | 'B' | 'C' | 'D';
  blackBoxDepth: 'none' | 'light' | 'moderate' | 'heavy';
  whiteBoxDepth: 'none' | 'light' | 'moderate' | 'deep';
  aiReviewRole: 'none' | 'supplement' | 'primary_reviewer' | 'auditor';
  requiredTestTypes: string[];
  estimatedEffort: string;
}

function selectTestingStrategy(
  risk: RiskLevel,
  familiarity: TechFamiliarity
): TestingStrategy {
  const isHighRisk = risk === 'high' || risk === 'critical';
  const knowsWell = familiarity === 'expert' || familiarity === 'author';

  // Quadrant A: High Risk, Know Tech Well
  if (isHighRisk && knowsWell) {
    return {
      quadrant: 'A',
      blackBoxDepth: 'moderate',
      whiteBoxDepth: 'deep',
      aiReviewRole: 'supplement',
      requiredTestTypes: [
        'unit tests (comprehensive)',
        'integration tests',
        'edge case tests',
        'race condition tests',
        'performance benchmarks',
        'security audit',
        'manual code review (you + AI)',
      ],
      estimatedEffort: 'High -- but you understand the code deeply'
    };
  }

  // Quadrant B: High Risk, Know Tech Poorly
  if (isHighRisk && !knowsWell) {
    return {
      quadrant: 'B',
      blackBoxDepth: 'heavy',
      whiteBoxDepth: 'light',
      aiReviewRole: 'primary_reviewer',
      requiredTestTypes: [
        'specification-driven black-box tests',
        'comprehensive input/output validation',
        'AI-generated edge case tests',
        'AI code review (explain implementation)',
        'AI self-critique sessions',
        'light manual scan for red flags',
        'property-based testing where possible',
      ],
      estimatedEffort:
        'Very High -- compensate lack of expertise with more testing'
    };
  }

  // Quadrant C: Low Risk, Know Tech Well
  if (!isHighRisk && knowsWell) {
    return {
      quadrant: 'C',
      blackBoxDepth: 'light',
      whiteBoxDepth: 'light',
      aiReviewRole: 'none',
      requiredTestTypes: [
        'smoke tests',
        'quick visual inspection (LGTM)',
        'basic happy-path tests',
      ],
      estimatedEffort: 'Low -- trust your expertise for low-stakes work'
    };
  }

  // Quadrant D: Low Risk, Know Tech Poorly
  return {
    quadrant: 'D',
    blackBoxDepth: 'moderate',
    whiteBoxDepth: 'none',
    aiReviewRole: 'supplement',
    requiredTestTypes: [
      'input/output validation',
      'data integrity checks',
      'basic sanity tests',
      '"Classic vibe coding" -- stay ignorant of internals',
    ],
    estimatedEffort: 'Low-Medium -- black-box is usually sufficient'
  };
}

// 使用範例
const strategy = selectTestingStrategy('high', 'novice');
console.log(`Quadrant ${strategy.quadrant}:`);
console.log(`Required tests: ${strategy.requiredTestTypes.join(', ')}`);
```

### 框架三：Code Accretion vs Code Destruction Pattern

```
  CODE ACCRETION (Clojure / Linux model) -- GOOD
  ================================================

  Time -->

  Year 1   |XXXXXXX|
  Year 2   |XXXXXXX|YYYY|
  Year 3   |XXXXXXX|YYYY|ZZZ|
  Year 4   |XXXXXXX|YYYY|ZZZ|WWWW|
  Year 5   |XXXXXXX|YYYY|ZZZ|WWWW|VVVV|

  Old code survives. APIs never break.
  Programs written a decade ago still work.
  This is what you want AI to do: ADD, not MODIFY.


  CODE DESTRUCTION (Scala model) -- BAD
  ================================================

  Time -->

  Year 1   |XXXXXXX|
  Year 2   |XXX    |YYYYYYYY|
  Year 3   |       |YYY     |ZZZZZZZ|
  Year 4   |       |        |ZZ     |WWWWWWWW|
  Year 5   |       |        |       |WW      |VVVVVVVVV|

  Old code deleted. APIs break constantly.
  Old programs don't compile anymore.
  This is what AI does if you don't stop it.


  GOLDEN RULE for AI:
  "You cannot break any existing functionality."
  -- Gene Kim (after learning the hard way)
```

```python
# ai_ci_check.py -- AI 增強的 CI/CD pipeline 檢查
from dataclasses import dataclass
from enum import Enum
from typing import Optional


class CheckSeverity(Enum):
    INFO = "info"
    WARNING = "warning"
    ERROR = "error"
    CRITICAL = "critical"


@dataclass
class CICheckResult:
    check_name: str
    severity: CheckSeverity
    message: str
    file_path: Optional[str] = None
    line_number: Optional[int] = None
    suggestion: Optional[str] = None


class AICIChecker:
    """
    AI-enhanced CI/CD checker
    模擬書中描述的 AI 增強 pipeline 檢查
    """

    def check_api_preservation(
        self, diff: str, existing_apis: list[str]
    ) -> list[CICheckResult]:
        """
        檢查是否有 API 被修改或刪除
        (書中的 "Don't torch your bridges" 原則)
        """
        results = []

        for api in existing_apis:
            if f"-{api}" in diff:  # 簡化的 diff 解析
                results.append(CICheckResult(
                    check_name="api_preservation",
                    severity=CheckSeverity.CRITICAL,
                    message=(
                        f"API '{api}' appears to be modified or removed. "
                        f"This violates the code accretion principle."
                    ),
                    suggestion=(
                        "Create a NEW function/endpoint instead of "
                        "modifying the existing one. "
                        "Prompt: 'You cannot break any existing "
                        "functionality.'"
                    )
                ))

        return results

    def check_module_boundaries(
        self, changed_files: list[str], module_map: dict[str, str]
    ) -> list[CICheckResult]:
        """
        檢查變更是否跨越模組邊界
        (書中的 minimize and modularize 原則)
        """
        results = []
        affected_modules = set()

        for f in changed_files:
            module = module_map.get(f, "unknown")
            affected_modules.add(module)

        if len(affected_modules) > 2:
            results.append(CICheckResult(
                check_name="module_boundary",
                severity=CheckSeverity.WARNING,
                message=(
                    f"Change spans {len(affected_modules)} modules: "
                    f"{affected_modules}. "
                    "This may indicate module boundary violation."
                ),
                suggestion=(
                    "Practice 'surgical commits': make the smallest "
                    "possible changes. "
                    "If cross-module changes are needed, "
                    "split into separate PRs."
                )
            ))

        return results

    def check_code_bloat(
        self,
        lines_added: int,
        lines_removed: int,
        task_description: str,
    ) -> list[CICheckResult]:
        """
        檢查 AI 是否產生了過多的程式碼
        (書中的 minimalism 原則)
        """
        results = []
        ratio = lines_added / max(lines_removed, 1)

        if lines_added > 500 and ratio > 10:
            results.append(CICheckResult(
                check_name="code_bloat",
                severity=CheckSeverity.WARNING,
                message=(
                    f"Added {lines_added} lines with only "
                    f"{lines_removed} removed (ratio: {ratio:.1f}x). "
                    "AI may have generated bloated code."
                ),
                suggestion=(
                    "Apply 'refactor after' pattern: accept "
                    "the functionality, "
                    "then instruct AI to refactor for conciseness. "
                    "Set a line budget for the task."
                )
            ))

        return results

    def check_branch_hygiene(
        self, branches: list[str], max_temp_branches: int = 5
    ) -> list[CICheckResult]:
        """
        檢查 branch litterbug 問題
        (Steve 失去一週程式碼的教訓)
        """
        results = []
        temp_branches = [
            b for b in branches
            if any(prefix in b for prefix in
                   ['temp-', 'codex:', 'fix-', 'wip-'])
        ]

        if len(temp_branches) > max_temp_branches:
            results.append(CICheckResult(
                check_name="branch_hygiene",
                severity=CheckSeverity.WARNING,
                message=(
                    f"{len(temp_branches)} temporary branches detected. "
                    "Risk of 'branch litterbug' syndrome. "
                    "Steve lost a week of work from unmerged branches."
                ),
                suggestion=(
                    "Review and prune temporary branches. "
                    "Verify all work is merged to main before deleting. "
                    "Add rule: every temp branch must be tracked "
                    "in the plan document."
                )
            ))

        return results
```

## 5. 最難理解的三個部分 (3 Hardest Sections)

### 難點 1：四象限 Testing Strategy 的實際應用判斷

- **為什麼難**：大多數工程師在同一個專案中會同時處於不同象限。你對 backend 是 "High Risk, Know Well" (Quadrant A)，但對 frontend 可能是 "High Risk, Know Poorly" (Quadrant B)。更困難的是，隨著 AI 幫你寫了越來越多你不熟悉的 code，你可能以為自己在 Quadrant A 但其實已經滑到了 Quadrant B。

- **白話解釋**：想像你是一個駕駛考官。你要判斷學生應該在哪種道路上練車。Quadrant A 是「你很熟悉的高速公路」——你知道每個彎道在哪裡，所以你可以做深入的白箱檢查。Quadrant B 是「你不熟悉的高速公路」——你不知道路況，所以你必須依賴更多的安全設備（black-box tests）和副駕駛（AI reviewer）。Quadrant D 是「你不認識的鄉間小路」——路況未知但風險低，只要車子還在動就行（basic input/output checks）。關鍵是：**你必須誠實評估自己在哪個象限**，而不是自我膨脹。

- **程式碼示範**：

```typescript
// quadrant-assessment.ts -- 誠實地評估你所在的象限

interface ProjectComponent {
  name: string;
  riskLevel: 'low' | 'high';
  techStack: string;
  yourExperience: 'none' | 'basic' | 'intermediate' | 'expert';
  aiGeneratedPercentage: number;  // 0-100
  hasExistingTests: boolean;
  usersAffected: number;
}

function assessQuadrant(component: ProjectComponent): {
  quadrant: 'A' | 'B' | 'C' | 'D';
  warning?: string;
  testingPlan: string[];
} {
  // 重要的誠實度調整：
  // 如果 AI 生成了 >60% 的 code，降低你的 familiarity 評估
  let effectiveExperience = component.yourExperience;
  if (component.aiGeneratedPercentage > 60) {
    const downgrade: Record<string, string> = {
      'expert': 'intermediate',
      'intermediate': 'basic',
      'basic': 'none',
      'none': 'none',
    };
    effectiveExperience =
      downgrade[component.yourExperience] as typeof component.yourExperience;
  }

  const isHighRisk = component.riskLevel === 'high'
    || component.usersAffected > 1000;
  const knowsWell = effectiveExperience === 'expert'
    || effectiveExperience === 'intermediate';

  let quadrant: 'A' | 'B' | 'C' | 'D';
  let testingPlan: string[];
  let warning: string | undefined;

  if (isHighRisk && knowsWell) {
    quadrant = 'A';
    testingPlan = [
      'Deep white-box review of all changed code paths',
      'Unit tests for every public function',
      'Integration tests for module boundaries',
      'Edge case and race condition tests',
      'Performance regression tests',
    ];
  } else if (isHighRisk && !knowsWell) {
    quadrant = 'B';
    testingPlan = [
      'Comprehensive black-box test suite from specifications',
      'Ask AI to explain every implementation decision',
      'Ask AI to identify its own edge cases and test them',
      'Property-based testing for data transformations',
      'Light manual scan for deleteAllData() or obvious red flags',
      'Run both old and new versions side-by-side if possible',
    ];
    warning = 'You are in the most challenging quadrant. '
      + 'Consider if the risk/reward tradeoff is worth it. '
      + 'Steve chose this for Wyvern TS client because '
      + 'it could replace 4 codebases.';
  } else if (!isHighRisk && knowsWell) {
    quadrant = 'C';
    testingPlan = [
      'Quick visual inspection (LGTM)',
      'Basic smoke tests',
      'Happy-path validation',
    ];
  } else {
    quadrant = 'D';
    testingPlan = [
      'Input/output validation (weigh beans before and after)',
      'Data integrity spot-checks',
      'Classic vibe coding: stay ignorant, check results',
    ];
  }

  if (effectiveExperience !== component.yourExperience) {
    warning = (warning || '') +
      ` NOTE: Your effective experience was downgraded from `
      + `"${component.yourExperience}" to "${effectiveExperience}" `
      + `because ${component.aiGeneratedPercentage}% of code is `
      + `AI-generated. You may not understand the internals as well `
      + `as you think.`;
  }

  return { quadrant, warning, testingPlan };
}

// 使用範例

  name: 'Payment Processing Module',
  riskLevel: 'high',
  techStack: 'TypeScript/Node.js',

  hasExistingTests: true,
  usersAffected: 50000,
});

```

### 難點 2：AI Code Bloat 與模組融合的隱性威脅

- **為什麼難**：AI 的 code bloat 問題不像一般的 bug——它不會讓你的程式 crash。它是漸進式的架構退化：每次 AI 新增功能時，它傾向於用最直接（但最不模組化）的方式實作。你要求一個 spinner，它可能把 spinner 邏輯直接嵌入到你的 API handler 中，而不是建立一個獨立的 UI component。單看 diff 時每個改動都「合理」，但累積效果是模組邊界的崩潰。

- **白話解釋**：想像你有一棟公寓大樓，每層樓都是獨立的——有自己的水管、電線、門。然後你請了一個超級快的裝修工（AI）。他每次裝修都很快、效果也好，但他有個壞習慣：他會把三樓的水管接到五樓的電線管道裡（因為「最短路徑」），把一樓的門鈴線路走七樓的天花板（因為「剛好有空間」）。每個單獨的裝修都「能用」，但現在你要換三樓的水管時，五樓的電就停了。更糟的是，你看裝修報告時每個改動都有道理——問題只在累積效應。

- **程式碼示範**：

```python
# module_fusion_detector.py -- 偵測模組正在融合的警示信號

from pathlib import Path
from collections import defaultdict
from typing import NamedTuple


class ImportEdge(NamedTuple):
    source_module: str
    target_module: str
    import_count: int


def analyze_module_coupling(
    project_root: str,
    module_dirs: list[str],
) -> dict:
    """
    分析模組間的耦合程度
    偵測 AI 是否正在破壞模組邊界
    """
    # 模擬分析結果 (實際實作會解析 AST)
    coupling_matrix: dict[str, dict[str, int]] = defaultdict(
        lambda: defaultdict(int)
    )

    # 健康的模組結構範例
    healthy_example = {
        "auth": {"auth": 15, "db": 2, "utils": 3},
        "payment": {"payment": 20, "db": 3, "utils": 2},
        "ui": {"ui": 25, "utils": 5},
    }
    # 注意：模組間的 import 很少，大部分 import 是模組內部的

    # AI 造成的退化結構範例
    degraded_example = {
        "auth": {"auth": 15, "db": 8, "payment": 6,
                 "ui": 4, "utils": 12},
        "payment": {"payment": 20, "db": 12, "auth": 9,
                    "ui": 7, "utils": 15},
        "ui": {"ui": 25, "auth": 5, "payment": 8,
               "db": 3, "utils": 20},
    }
    # 注意：跨模組 import 大量增加 -- 模組正在融合!

    return degraded_example


def calculate_fusion_score(
    coupling: dict[str, dict[str, int]]
) -> dict:
    """
    計算模組融合分數
    分數越高表示模組邊界越模糊
    """
    results = {}

    for module, imports in coupling.items():
        internal = imports.get(module, 0)
        external = sum(
            count for mod, count in imports.items()
            if mod != module
        )
        total = internal + external

        # Fusion ratio: 外部依賴 / 總依賴
        # 健康: < 0.3 (大部分 import 在模組內)
        # 警告: 0.3 - 0.5
        # 危險: > 0.5 (超過一半的 import 來自其他模組)
        fusion_ratio = external / total if total > 0 else 0

        results[module] = {
            "internal_imports": internal,
            "external_imports": external,
            "fusion_ratio": round(fusion_ratio, 2),
            "status": (
                "HEALTHY" if fusion_ratio < 0.3
                else "WARNING" if fusion_ratio < 0.5
                else "FUSING -- IMMEDIATE ACTION NEEDED"
            ),
            "recommendation": (
                "Module boundaries intact."
                if fusion_ratio < 0.3
                else "Review recent AI changes for boundary violations."
                if fusion_ratio < 0.5
                else "STOP adding features. Refactor module boundaries "
                     "BEFORE continuing. Gene's 3-day rebuild "
                     "started from ignoring this signal."
            )
        }

    return results


# 範例輸出
coupling = analyze_module_coupling("/project", ["auth", "payment", "ui"])
scores = calculate_fusion_score(coupling)

for module, score in scores.items():
    print(f"\n{module}:")
    print(f"  Fusion ratio: {score['fusion_ratio']}")
    print(f"  Status: {score['status']}")
    print(f"  Action: {score['recommendation']}")

# Expected output for degraded example:
# auth:
#   Fusion ratio: 0.67
#   Status: FUSING -- IMMEDIATE ACTION NEEDED
#   Action: STOP adding features. Refactor module boundaries...
```

### 難點 3：在組織層面推動流程改革以支撐 Vibe Coding

- **為什麼難**：這不是技術問題，而是組織政治問題。書中提到 GitLab 的 Jessie Young 的案例：即使你有 10x 的 AI 生產力提升，如果每次 push to production 需要 8 個人簽核（為了 SOC 2 compliance），你的速度瓶頸不在 coding 而在 approval。Morgan Stanley 花了大量心力用 ML model 證明自動化審批比人工審批更安全——這不是因為技術困難，而是因為你必須用數據說服 risk management 和 compliance team。

- **白話解釋**：想像你買了一輛法拉利（AI coding 能力），但你住在一個限速 30 公里的社區裡（組織流程）。法拉利的引擎再好，你也只能開 30。要讓法拉利發揮價值，你有兩個選擇：(1) 搬到沒有速限的地方（換公司），或 (2) 說服社區管委會提高速限——但你需要數據證明「更快的速度其實更安全」（Morgan Stanley 的做法：AI 審批的 0% incident rate vs 人工的 1.5%）。大部分工程師只關注「怎麼把車開更快」，但真正的瓶頸是道路規則。

- **程式碼示範**：

```typescript
// process-bottleneck-analyzer.ts
// 分析組織流程中的 vibe coding 瓶頸

interface DeploymentProcess {
  step: string;
  avgDuration: { hours: number };
  isAutomatable: boolean;
  requiresHumanApproval: boolean;
  complianceRequirement?: string;
}

interface BottleneckAnalysis {
  totalLeadTime: number;  // hours
  codingTime: number;
  waitingTime: number;
  bottleneckStep: string;
  speedupWithAI: number;  // multiplier
  actualSpeedup: number;  // after bottleneck
  recommendation: string;
}

function analyzeDeploymentBottleneck(
  steps: DeploymentProcess[]
): BottleneckAnalysis {
  const codingSteps = steps.filter(s =>
    s.step.includes('code') || s.step.includes('test'));
  const waitingSteps = steps.filter(s => s.requiresHumanApproval);

  const totalLeadTime = steps.reduce(
    (sum, s) => sum + s.avgDuration.hours, 0
  );
  const codingTime = codingSteps.reduce(
    (sum, s) => sum + s.avgDuration.hours, 0
  );
  const waitingTime = waitingSteps.reduce(
    (sum, s) => sum + s.avgDuration.hours, 0
  );

  // 找到最大瓶頸
  const bottleneck = steps.reduce((max, s) =>
    s.avgDuration.hours > max.avgDuration.hours ? s : max
  );

  // AI 能加速 coding 10x，但無法加速 approval
  const aiCodingTime = codingTime / 10;
  const newTotal = aiCodingTime + waitingTime;

  return {
    totalLeadTime,
    codingTime,
    waitingTime,
    bottleneckStep: bottleneck.step,
    speedupWithAI: codingTime / aiCodingTime, // theoretical
    actualSpeedup: totalLeadTime / newTotal,   // actual
    recommendation: waitingTime > codingTime
      ? `Process bottleneck (${waitingTime}h waiting vs `
        + `${codingTime}h coding). `
        + `Follow Morgan Stanley's approach: use ML model to prove `
        + `automated approvals have LOWER incident rate than human `
        + `reviews. Their result: 0% vs 1.5% incident rate.`
      : `Technical bottleneck. AI coding speedup will have `
        + `direct impact on lead time.`
  };
}

// 模擬 GitLab-style 8-approval process
const gitlabProcess: DeploymentProcess[] = [
  { step: 'Write code', avgDuration: { hours: 4 },
    isAutomatable: true, requiresHumanApproval: false },
  { step: 'Write tests', avgDuration: { hours: 2 },
    isAutomatable: true, requiresHumanApproval: false },
  { step: 'Code review #1', avgDuration: { hours: 8 },
    isAutomatable: false, requiresHumanApproval: true },
  { step: 'Code review #2', avgDuration: { hours: 8 },
    isAutomatable: false, requiresHumanApproval: true },
  { step: 'Security review', avgDuration: { hours: 24 },
    isAutomatable: false, requiresHumanApproval: true,
    complianceRequirement: 'SOC 2' },
  { step: 'QA approval', avgDuration: { hours: 16 },
    isAutomatable: false, requiresHumanApproval: true },
  { step: 'Deploy', avgDuration: { hours: 1 },
    isAutomatable: true, requiresHumanApproval: false },
];

const analysis = analyzeDeploymentBottleneck(gitlabProcess);
console.log(`Total lead time: ${analysis.totalLeadTime}h`);
console.log(`Coding: ${analysis.codingTime}h`);
console.log(`Waiting: ${analysis.waitingTime}h`);
console.log(`AI coding speedup: ${analysis.speedupWithAI}x`);
console.log(`Actual speedup: ${analysis.actualSpeedup.toFixed(1)}x`);
console.log(`Recommendation: ${analysis.recommendation}`);

// Output:
// Total lead time: 63h
// Coding: 6h, Waiting: 56h
// AI coding speedup: 10x (theoretical)
// Actual speedup: 1.1x (!!!)
// Recommendation: Process bottleneck...
// The 10x AI speedup only saves 5.4h out of 63h total!
```

### 六大架構啟示

1. **API Contracts are Sacred**：「You cannot break any existing functionality」應該被刻在每個 AGENTS.md 裡。Code accretion (Clojure/Linux 模式) 遠優於 code destruction (Scala 模式)。AI 時代的最佳藉口——「維護 backward compatibility 太費工」——已經不成立了，因為 AI 可以幫你維護。

2. **Testing Strategy Must Match Your Quadrant**：不是所有 code 都需要同等程度的 testing。用四象限矩陣誠實評估你的位置，然後選擇對應的策略。特別注意：當 AI 生成了大部分 code 時，你的 effective familiarity 會降低，可能需要升級 testing rigor。

3. **Workspace Discipline Prevents Catastrophe**：Steve 在三週內經歷了三次大型 workspace 事故。Partition (分區)、Label (標籤)、Simplify (簡化) 是三大防禦策略。Terminal window 顏色編碼不是花俏——它是存活機制。

4. **CI/CD is Your Last Line of Defense**：AI 增強的 CI/CD pipeline 可以做到人工無法做到的事：每次 commit 都有 security review、guideline enforcement、intelligent error explanation。但注意成本控制——不要讓 10 個 agent 在每次 build 時燒 token。

5. **Git Discipline is Survival**：Push often、prune branches carefully、keep Git commands in your own hands (至少在 AI 可信度提高之前)。Steve 的教訓：一個未 merge 到 main 的 branch 就是一個定時炸彈。

6. **Process is the Real Bottleneck**：10x coding 速度在 8-approval 流程面前毫無意義。用 Morgan Stanley 的方法：用數據證明自動化比人工更安全，然後建立 "fast lane"。這是真正的 organizational FAAFO。

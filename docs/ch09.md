## 1. 概述 (Overview)

本章從玩具範例跨入真實工程問題，系統性地介紹 vibe coding 的四層工具光譜——從 chatbot 到 coding agent 到多 agent 協作。透過 Gene 的 video excerpter 和 Steve 的 Puppeteer 兩個 war story，展示了一個關鍵洞見：**AI 能直接執行並驗證自己的產出時，生產力提升從 2x 躍升到 10x。** 本章還提煉出 vibe coding 的核心實踐：對話式互動、讓錯誤自己說話、善用 AI 百科全書式知識、從模糊到精確的迭代。

## 2. 學習目標 (Learning Objectives)

完成本章後，你應該能夠：

- **辨識** vibe coding loop 的七個步驟，並判斷何時該從 agent 降級到 chat
- **理解** 為什麼 tool access（特別是 MCP）是 coding agent 生產力的決定性因素
- **實作** 從「模糊的需求描述」到「精確的 AI 可執行指令」的轉化過程
- **應用** 「對話而非合約」的互動原則，在實際專案中保持 FAAFO 節奏

## 3. 章節大綱 (Outline)

### 3.1 Vibe Coding Loop（七步循環）
- Frame → Decompose → Converse → Review → Test → Refine → **Automate**
- 第七步「自動化你的工作流」是進階關鍵
- 與傳統開發循環的差異：每一步都不能鬆懈

### 3.2 War Story：Gene 的 Video Excerpter
- 目標：將 15 年的 YouTube 截圖轉為可分享的字幕影片片段
- 任務拆解：影片擷取 → 字幕處理 → SRT 生成 → 字幕疊加
- 47 分鐘完成，使用 chat-based vibe coding
- 教訓：AI 生成的測試案例本身可能是錯的

### 3.3 Coding Agents 的躍進
- Gene 的 Trello API 故事：從「人肉 curl 打手」到讓 agent 自寫 Clojure 程式執行
- Steve 的 Puppeteer 故事：讓 agent「看見」UI，從 Pin the Tail on the Donkey 到 10x 加速
- 核心原則：**給 AI 工具 = 讓 AI 關上 feedback loop**

### 3.4 工具選擇策略
- 預設選 agent → 需要時降級到 chat → 最後手動介入
- 保持 escape hatch：避免 one-way door
- Cambrian explosion of coding interfaces

### 3.5 Vibe Coding 核心實踐
- Conversations, not commands or contracts
- Letting errors speak for themselves
- Leaning into AI's encyclopedic knowledge
- From vague to precise: sharpening requests

## 5. 最難理解的三個部分 (3 Hardest Sections)

### 難點 1：為什麼「讓 agent 自己跑測試」是質的飛躍？

- **為什麼難**：表面上只是「少了一步 copy-paste」，但實際影響遠大於此。很多 RD 還在用 chat 模式工作，沒有意識到自己是 bottleneck。
- **白話解釋**：想像你是翻譯官，站在兩個說不同語言的人中間。你每句話都要轉述，對話速度取決於你。現在想像兩個人突然學會了對方的語言——你就不再是瓶頸了。Agent + tool access 就是讓 AI 直接和開發環境「說同一種語言」。
- **程式碼示範**：

```typescript
// 情境：修復 API 錯誤

// === Chat 模式（你是 middleware）===
// 1. 你跑 curl，得到 401 error
// 2. 你 copy error 給 AI
// 3. AI 建議改 header
// 4. 你手動改 curl 指令再跑
// 5. 又 fail，你再 copy 給 AI
// 6. 重複 N 次... (Gene 花了 45 分鐘)

// === Agent 模式（AI 自己關閉 loop）===
// Agent prompt:
// "寫一個測試腳本來診斷 Trello API 的 auth 問題，
//  嘗試不同的認證方式直到找到能用的。"

// Agent 自行執行:
async function diagnoseAuth() {
  const methods = [
    { name: 'query-params', fn: tryQueryParams },
    { name: 'basic-auth', fn: tryBasicAuth },
    { name: 'bearer-token', fn: tryBearerToken },
    { name: 'oauth-header', fn: tryOAuthHeader },  // <-- 45 秒找到
  ];

  for (const method of methods) {
    const result = await method.fn();
    console.log(`${method.name}: ${result.status}`);
    if (result.status === 200) {
      console.log(`SUCCESS: Use ${method.name}`);
      return method;
    }
  }
}
// Agent 在 45 秒內完成 Gene 花 45 分鐘做的事
```

### 難點 2：「對話而非合約」聽起來很不工程

- **為什麼難**：受過嚴格訓練的工程師會本能地抗拒「隨便聊聊就好」的說法。Prompt engineering 的嚴謹性看起來才是正確方向。
- **白話解釋**：Prompt engineering 像寫合約——每個字都要精雕細琢，因為合約簽完就不能改了。Vibe coding 像 Slack 對話——你可以隨時補充、修正、重新表述。合約適合 production prompt template，Slack 適合解決當下的問題。兩者不衝突，但大多數日常開發工作更適合後者。
- **實務對比**：

```python
# Prompt Engineering 風格（適合 reusable template）：
prompt_template = """
You are a senior Python developer specializing in FastAPI.
Given the following database schema: {schema}
Generate a REST endpoint that:
1. Accepts POST requests at /api/v2/transactions
2. Validates input against the TransactionCreate Pydantic model
3. Returns 201 with the created resource
4. Handles duplicate key errors with 409
5. Logs all operations using structured logging
"""

# Vibe Coding 風格（適合解決當下問題）：
# "這個 endpoint 回 500 了，看一下 stack trace"
# [貼上 stack trace]
# "對，就是那個 duplicate key 的問題，修一下"
# "順便加個 log"
# "好了，跑測試看看"

# 兩種風格各有適用場景，不是互斥的
```

### 難點 3：第七步「自動化你的工作流」到底該自動化什麼？

- **為什麼難**：作者說「任何 friction 都是巨大的機會成本」，但沒有具體列出該自動化的清單。對忙碌的 RD 來說，「自動化工作流」本身就需要時間投入。
- **白話解釋**：就像整理工具箱。每次做木工都要花五分鐘找鉗子，看起來不嚴重。但如果你每天做十次，一年就浪費了 300 小時。自動化就是把鉗子固定放在隨手可及的地方。
- **具體建議**：

```typescript
// 應該自動化的事情（按優先級排序）：

// 1. 錯誤訊息的餵入 → 用 coding agent 直接看 terminal
//    自動化前：手動 copy error → paste to AI → copy fix → apply
//    自動化後：agent 直接讀 terminal output

// 2. 測試執行 → 設定 agent 可以跑 test suite
//    自動化前："幫我寫測試" → copy code → run → copy result
//    自動化後："寫測試然後跑，修到全過為止"

// 3. Context 準備 → 用 CLAUDE.md / .cursorrules
//    自動化前：每次對話都要解釋 project 結構
//    自動化後：agent 自動讀取 project context

// 4. 常用指令 → 寫成 shell script 或 MCP tool
//    例如：
//    - deploy-staging.sh
//    - run-integration-tests.sh
//    - generate-api-docs.sh

// 5. Code review checklist → 寫進 system prompt
//    自動化前：每次 review 都要提醒同樣的事
//    自動化後：AI 自動按 checklist 檢查
```

## 6. 本章小結與架構啟示

### 對資深 RD 的核心洞見

**Gene 的 Video Excerpter 故事真正的教訓不是「AI 很厲害」，而是「以前不可能做的事，現在可以做了」。** 這就是 FAAFO 的精髓——Ambitious 的 A。當開發成本趨近於零時，過去因 ROI 不足而放棄的專案變得值得嘗試。

### 工具選擇的架構思維

```
決策樹：選擇正確的 AI 工具

需要修改檔案或執行指令？
├── 是 → 用 Coding Agent (Claude Code / Codex / Amp)
│     │
│     ├── Agent 能直接存取需要的工具？
│     │   ├── 是 → 全自動模式，你只做 review
│     │   └── 否 → 找 MCP 或寫 wrapper script
│     │           └── 還是不行 → 降級到 chat，你當 middleware
│     │
│     └── Agent 反覆犯錯？
│         ├── 降級到 chat，手動引導
│         └── 或開新對話，重新 frame 問題
│
└── 否 → 用 Chatbot (Claude / ChatGPT / Gemini)
      ├── 需要深度分析 → 用 browser 版大模型
      ├── 隨手問問 → ChatGPT Voice Mode
      └── 需要搜尋 → 開啟 web search 功能
```

### 架構師最該記住的一句話

> "Type less; lean on AI more." — Steve Yegge

這不只是效率建議，而是架構思維的轉變。每一次你手動做某件事，問自己：

1. 這件事能不能讓 agent 做？
2. 如果不能，是因為 agent 缺少什麼工具？
3. 能不能透過 MCP 把那個工具接上去？

這個思考框架會讓你的 vibe coding 生產力持續攀升。

### 從 War Stories 提取的反模式

| 反模式 | 症狀 | 解法 |
|--------|------|------|
| Human as Middleware | 你在 AI 和工具之間不停 copy-paste | 給 agent tool access |
| Pin the Tail on the Donkey | 你在描述 UI 問題，AI 看不到畫面 | 接 Puppeteer / screenshot MCP |
| Chat 退化 | Agent 要求你執行指令，變成 chat session | 讓 agent 用替代方式自己執行 |
| Prompt 完美主義 | 花太多時間修改 prompt 而非看結果 | 對話式迭代，關注 output |

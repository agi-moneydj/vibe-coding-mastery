## 1. 概述 (Overview)

當 vibe coding 從個人實驗變成團隊標準作業時，你需要一本「協作食譜」——讓每一個加入你廚房的人（不管是人類還是 AI）都知道東西放在哪裡、該用什麼做法、以及如何避開常見地雷。本章提出了一套三層式的標準體系（組織級 → 團隊級 → 專案級），並且用 Toyota 的「standard work」和 Google 的 monorepo 實踐來說明為什麼「活的文件」比「死的規範」重要。此外，本章深入探討了 Dr. Daniel Rock 的「The Drift」概念——AI 如何大幅降低協調成本，以及 Dr. Matt Beane 關於「隱藏創新者」和新興角色的研究。

## 3. 章節大綱 (Outline)

### 3.1 協作食譜：建立共享 AI 規則與標準

- **基礎原則**：先定義什麼是「great」，然後寫下來
- **三層式標準體系**：
  - **Org-level**：安全底線（sanitize inputs、不 commit secrets、parameterized queries）
  - **Team-level**：命名規範、test framework 選擇、code style
  - **Project-level**：具體工具選擇（Vitest for unit tests、tabs in build scripts）
- **Standard Work（Toyota 概念）**：當前最佳做法，持續透過經驗改善
- **Google monorepo 的智慧**：想知道怎麼用某個 library？搜尋 repo 看別人怎麼用
- **Rules files 的維護**：花每天「顯著比例」的時間來 curate prompt files、Markdown plans、agent context
- **讓 AI 幫忙維護 rules**：餵它你現有的 Markdown / database，要求更精簡版本
- **存儲位置建議**：repo 中的 `/ai-rules/org`、`/ai-rules/team`、`/ai-rules/project`

### 3.2 心靈融合與 AI 副廚：降低協調成本

- **「The Drift」概念**（Dr. Daniel Rock，Wharton School，源自電影 Pacific Rim）
- **Dr. Rock 的 GitHub Apps 故事**：
  - 週二：Google search + 空白 Markdown doc + Claude chatbot → 8 個深入問題後 spec 成形
  - 週三：chief-of-staff（designer）加入同一文件畫 UI flow
  - 週四：senior developer 加入 Cursor 開始建 tests 和 scaffolding
  - 48 小時：從第一次 Google 搜尋到 working GitHub App
- **為什麼快**：消除了兩種隱形成本
  - Connection setup time（人與人之間的握手時間）
  - Bandwidth mismatch（溝通頻寬不對稱）
- **歷史類比**：從 mechanical driveshaft 到電力 → Layer 2 創新解鎖 Layer 3 新布局
- **自主不等於孤立**：Autonomous = unblocked，不是 isolated

### 3.3 軟體的潛在新角色

- Dr. Matt Beane 的研究：自動化如何重塑工作
- **倉庫機器人的故事**：不會說英文的工人建議用 icon 取代 error message → 珍貴的 UX 洞見
- **RHLF-trained 機器人 startup**：招「喜歡打電動的人」當操作員 → 他們升級為 UX/data science/mechatronics
- **七個新興角色**：
  1. **Product Prototyper**：PM 自己交 prototype 給開發者
  2. **Platform Designer**：1/3 PM + 1/3 designer + 1/3 infra engineer
  3. **Fleet Fixer**：監督多個 AI 系統，必要時介入
  4. **Agent Expert**：建立和維護 domain-specific agents
  5. **Fleet Supervisor**：指揮多 AI 系統、設計互動結構
  6. **Platform Engineer**：往上推 stack，讓更多 fuzzy stuff 可以安全通過 sandbox
  7. **Apex Builder**：把 vibe-coded prototype 打造成堅固產品；修復 AI migration 留下的 legacy mess

### 3.4 CS 課程的潛在變革

- **Code Reading**：每天可能要讀數千行 code，需要 speed-reading 訓練
- **Precise Communication**：AI 時代的成功取決於你能多有效地指揮 AI → writing is thinking
- **Concentration / Multi-project**：同時管理多個 agent 的認知負荷 + version control discipline
- **Software Modularity & Architecture**：比語言細節更重要
- **Entrepreneurial Awareness**：小團隊 + AI 可以顛覆市場

## 4. 關鍵框架與公式 (Key Frameworks)

### 框架一：三層式 AI Rules 階層

**何時使用**：當你要從零開始建立團隊的 AI 協作標準時。

```
 /ai-rules/
 |
 +-- /org/                          (組織級: 安全底線)
 |   +-- security.md                - 永不 commit secrets
 |   +-- code-quality.md            - sanitize all inputs
 |   +-- testing.md                 - tests before commits
 |   +-- ai-usage-policy.md         - 授權模型清單 + 成本限制
 |
 +-- /team/                         (團隊級: 慣例與風格)
 |   +-- naming-conventions.md      - camelCase for TS, snake_case for Python
 |   +-- review-process.md          - PR 必須有 1 human + 1 AI review
 |   +-- framework-choices.md       - React + FastAPI + PostgreSQL
 |   +-- agent-guidelines.md        - 如何寫有效的 agent prompt
 |
 +-- /project/                      (專案級: 具體工具與做法)
     +-- AGENTS.md                  - 本專案的 AI context 與指令
     +-- testing-strategy.md        - Vitest for units, Playwright for E2E
     +-- api-patterns.md            - retry wrapper, error format, auth flow
     +-- architecture-decisions.md  - ADR (Architecture Decision Records)

 MAINTENANCE RULE:
 - 每天花 15-30 分鐘 curate 這些文件
 - 讓 AI 幫你精簡：「這個 rules file 有哪些過時的部分？」
 - Version control everything: 規則的變更也是 PR

 ANALOGY: Toyota's "Standard Work"
 = 「目前為止最好的做法，但預期會持續改善」
 ≠ 「刻在石頭上不能動的規範」
```

### 框架二：The Drift — 心靈融合工作模式

**何時使用**：當跨職能小團隊需要在 48 小時內從零到一建出 working prototype 時。

```
 TRADITIONAL HANDOFF MODEL (串行):
 +--------+     +--------+     +--------+     +--------+
 | PM     |---->| Design |---->| Dev    |---->| QA     |
 | writes |     | draws  |     | builds |     | tests  |
 | PRD    |     | UI     |     | code   |     |        |
 +--------+     +--------+     +--------+     +--------+
  Week 1         Week 2         Week 3-4       Week 5
  Total: 5 weeks, 4 handoffs, N misunderstandings

 THE DRIFT MODEL (融合):
 +--------------------------------------------------+
 |              SHARED MARKDOWN DOCUMENT             |
 |  (part chat log, part design doc, part code       |
 |   review transcript)                              |
 |                                                   |
 |  Day 1: PM prompts Claude -> spec takes shape     |
 |  Day 2: Designer joins same doc -> UI flows       |
 |  Day 3: Dev joins Cursor -> tests + scaffolding   |
 |                                                   |
 |  EVERYONE is writing prompts, code, and tests     |
 |  in real time -> 3x developer capacity            |
 +--------------------------------------------------+
  Total: 2 days, 0 handoffs, shared understanding

 WHY IT WORKS:
 1. Zero "connection setup time" (no greeting/aligning/clarifying)
 2. Zero "bandwidth mismatch" (everyone sees the same document)
 3. The living doc IS the spec, design, and implementation
 4. AI acts as universal translator between disciplines

 HISTORICAL PARALLEL:
 Mechanical driveshaft -> Electric motors
 = Machines DECOUPLED from central shaft
 = Any machine can go ANYWHERE
 = Radical new factory layouts become possible

 Vibe Coding -> The Drift
 = Work DECOUPLED from rigid handoff chains
 = Any team member can contribute ANYWHERE
 = Radical new team structures become possible
```

## 5. 最難理解的三個部分 (3 Hardest Sections)

### 難點 1：Standard Work 的「活文件」哲學 — 如何避免 Rules Rot

- **為什麼難**：大多數工程團隊都有一個放了三年沒人看的 `CONTRIBUTING.md`。作者說要「每天花顯著比例的時間 curate 你的 rules files」——這聽起來像是在要求已經很忙的工程師做更多行政工作。而且，怎麼判斷一條規則是「已過時」還是「仍然重要但大家忘了」？
- **白話解釋**：想像你開餐廳，菜單上還寫著「季節限定：草莓甜點」——但現在是冬天。客人不會因為菜單寫了就覺得合理。你的 rules file 也一樣：過時的規則比沒有規則更糟糕，因為它會讓 AI（和新人）產出不合時宜的東西，而且大家會養成「反正 rules file 不準的，不用看」的習慣。所以維護不是 nice-to-have，是 critical path。
- **程式碼示範**：

```python
# 自動化 rules file 健康度檢查

import os
import datetime
from pathlib import Path

class RulesHealthChecker:
    """
    每週自動檢查 /ai-rules/ 目錄中的文件健康度
    可以整合到 CI pipeline 或用 cron job 執行
    """

    STALE_THRESHOLD_DAYS = 30  # 超過 30 天沒更新就是 warning

    def check_staleness(self, rules_dir: str) -> list[dict]:
        """掃描所有 rules files，標記過時的"""
        issues = []
        for md_file in Path(rules_dir).rglob("*.md"):
            last_modified = datetime.datetime.fromtimestamp(
                os.path.getmtime(md_file)
            )
            age_days = (datetime.datetime.now() - last_modified).days

            if age_days > self.STALE_THRESHOLD_DAYS:
                issues.append({
                    "file": str(md_file),
                    "last_modified": last_modified.isoformat(),
                    "age_days": age_days,
                    "severity": "warning" if age_days < 90 else "critical",
                    "suggested_action": (
                        f"Review and update, or explicitly confirm "
                        f"this rule is still valid (touch the file)."
                    ),
                })
        return issues

    def check_consistency(self, rules_dir: str) -> list[dict]:
        """檢查 org/team/project 層級之間是否有矛盾"""
        # 例如：org-level 說 "use parameterized queries"
        # 但 project-level 有 raw SQL pattern
        # 這需要 AI 來做語義分析
        pass

    def generate_report(self, rules_dir: str) -> str:
        stale = self.check_staleness(rules_dir)
        if not stale:
            return "All rules files are up-to-date."

        report = f"## Rules Health Report\n\n"
        report += f"Found {len(stale)} stale files:\n\n"
        for issue in stale:
            report += (
                f"- **{issue['file']}** "
                f"({issue['age_days']} days old, "
                f"{issue['severity']})\n"
            )
        return report


# 整合到 CI: 在每次 PR merge 時檢查
# .github/workflows/rules-health.yml:
# - name: Check AI rules freshness
#   run: python check_rules_health.py /ai-rules/
```

### 難點 2：The Drift 為什麼可行？— 打破 Handoff 的經濟學

- **為什麼難**：大多數人的工作經驗是「分工 → 交接 → 下一步」。The Drift 模式要求多人同時在同一個 artifact 上工作，這違反了我們對「避免衝突」的直覺。而且，如果 PM、designer、developer 都在同一個 Markdown 文件裡寫 prompts，那不是更容易出混亂嗎？
- **白話解釋**：想像三個人一起拼一幅拼圖。傳統做法是：A 拼完邊框 → 交給 B 拼天空 → 交給 C 拼地面。每次交接，新的人都要花時間「搞清楚目前進度」。The Drift 的做法是：三個人同時圍在桌邊拼。A 說「我覺得這塊是天空的」，B 馬上接「不，那是湖的倒影，你看這裡」——資訊立即流動，zero handoff。AI 在這裡的角色是一個永遠在場的翻譯官：它把 PM 的「使用者需要...」翻譯成 developer 的「endpoint spec 是...」，即時且無損。
- **實務範例**：

```typescript
// The Drift 的文件結構 — 一個 living Markdown 即是 spec/design/code

/*
# Feature: Customer Repository Data Extraction

## Goals (PM wrote this, Day 1)
- Extract repo metadata via GitHub Apps API
- Support org-level and repo-level permissions
- < 5 second response time for 95th percentile

## Questions from AI (Claude asked these, Day 1)
1. What GitHub App permissions are needed?
   -> PM answered: read-only for repo metadata, issues, PRs
2. Do we need webhook support?
   -> PM: Not for v1, but design for extensibility
... (8 questions total)

## UI Flow (Designer added this, Day 2)

- Data display: table with sortable columns
- Error states: retry button + support link

- Data: PostgreSQL for caching, 5-min TTL

## Tests (Dev + AI wrote these, Day 3)

## Status
- [x] Spec finalized (Day 1)

### 難點 3：新興角色的辨識 — 你的「隱藏創新者」在哪裡？

- **為什麼難**：Dr. Beane 的研究發現了一個令人不安的模式：最有價值的實務創新往往來自組織底層（junior、non-traditional 背景的人），但這些創新經常被忽視、被主管搶功、或直接消失。在你的組織中，可能正有人在 quietly experimenting with AI，發現了極有價值的東西——但你不知道。
- **白話解釋**：想像一個自動化倉庫，有個夜班工人不會說英文，但她發現機器人的 error message 換成 icon 會讓所有人都能看懂。這個洞見比任何 UX 研究都來得實際和有價值。但如果沒有人主動去問「夜班的人有沒有發現什麼？」，這個洞見就會消失。你的軟體團隊也是一樣——可能有個 junior 工程師在用 AI 做了某個巧妙的事情，但因為「junior 的意見不重要」所以沒人知道。
- **Leader 的行動清單**：

```
 HOW TO FIND YOUR HIDDEN INNOVATORS:

 +---+------------------------------------------+
 | 1 | 定期問團隊：「最近用 AI 做了什麼                |
 |   | 讓你驚訝的事？」（不只問 senior）            |
 +---+------------------------------------------+
 | 2 | 建立 channel 讓任何人都能分享 AI tips        |
 |   | （匿名也行，降低分享門檻）                    |
 +---+------------------------------------------+
 | 3 | 追蹤 token burn 的 outliers                |
 |   | （特別是 title 與 usage 不成比例的人）       |
 +---+------------------------------------------+
 | 4 | 每月一次「AI Show & Tell」                  |
 |   | （15 min lightning talks，任何人都能報名）   |
 +---+------------------------------------------+
 | 5 | 對發現有價值做法的人給予正式 credit           |
 |   | （防止主管搶功的文化保護）                    |
 +---+------------------------------------------+
 | 6 | 為「沒有名字的角色」創造 title               |
 |   | （像 startup 為 game players                |
 |   |  創造 robot operator title）                |
 +---+------------------------------------------+

## 6. 本章小結與架構啟示

**最重要的一句話**：Standard work 不是官僚——它是讓每一個 AI chef（和每一個新 human teammate）走進你的廚房時，就立刻知道刀放在哪裡、火候怎麼調、出了問題找誰的機制。

**架構啟示**：

1. **Rules files 是你最重要的 shared artifact**。投入在 `/ai-rules/` 目錄的時間，其 ROI 比你花在任何單一 feature 上的時間都高。因為好的 rules files 會讓你的所有 agent 和所有 teammate 都更有效率——這是一個 multiplicative 而非 additive 的效果。

2. **The Drift 是對 waterfall/agile handoff 的根本挑戰**。如果你的團隊還在用「PM 寫 spec → 丟給 designer → 丟給 developer」的流程，AI 帶來的加速效果會被 handoff 的等待時間完全抵消。嘗試讓 2-3 個人圍繞一個 shared Markdown document 同時工作，AI 作為即時翻譯官。

3. **「電氣化」類比值得深思**。工廠從 driveshaft 轉到電力時，不只是換了動力來源——而是整個工廠 layout 重新設計了，因為機器不再需要排在同一根軸上。Vibe coding 也是一樣：不只是把 AI 插入現有流程，而是要重新思考工作如何被分解和整合。

4. **隱藏創新者是你最大的未開發資產**。Dr. Beane 的倉庫研究和 RHLF startup 的故事都指向同一個結論：最有價值的實務知識往往來自離問題最近的人，而不是層級最高的人。主動去找他們，給他們舞台和 credit。

5. **CS 教育的四大支柱正在重排**。Code reading > code writing，Communication > algorithm memorization，Architecture > language specifics，Entrepreneurship > specialization。如果你還在用 2015 年的標準招人和培訓人，你會錯過 AI 時代最需要的人才類型。

# 第 18 章：建立 Vibe Coding 文化 (Creating a Vibe Coding Culture)

## 章節精通指南 | Chapter Mastery Guide

> **書籍**：Vibe Coding: Building Production-Grade Software with AI
> **作者**：Gene Kim & Steve Yegge
> **導讀視角**：資深軟體架構師
> **程式碼範例**：TypeScript / Python

---

## 1. 概述 (Overview)

本章從個人技能轉向組織變革：你個人已經掌握 vibe coding，但怎麼讓整個團隊、整個公司都跟上？作者提出一套系統性的領導策略——從「自己先下廚」到「點燃可見的火焰」、從 Sourcegraph 的 token burn leaderboard 到 Malcolm Gladwell 的 tipping point 三角（Mavens、Connectors、Salespeople）。同時，本章也直面 AI 時代招聘標準的根本轉變：溝通能力、大量 code review 能力、以及 AI interaction proficiency 已經取代了傳統的「語言精通度」和「演算法背誦」。

---

## 2. 學習目標 (Learning Objectives)

完成本章後，你應該能夠：

- **辨識** 你組織中的 Mavens（技術探索者）、Connectors（知識傳播者）、Salespeople（熱情推動者），並知道如何借力他們推動 AI 採用
- **理解** 為什麼「領導者親自動手」比任何分析師報告都更有效，以及 token burn 作為 KPI 的邏輯
- **實作** 一套安全的 AI rollout 計畫——從 pilot team 到全組織，包含 guardrails、blameless post-mortems、以及 hero story 策略
- **設計** 適合 AI 時代的面試流程：評估 prompt engineering 能力、code review at scale 能力、以及 AI interaction 的現場測試

## 3. 章節大綱 (Outline)

### 3.1 領導者必須做的事：高管策略

- **自己先開火**：在要求團隊之前，先花 10 小時親手使用 AI
- **點燃火焰**：公開分享自己的 FAAFO 成果，展現可見的樂觀
- **Token Burn KPI**：Dr. Matt Beane 的洞見 — tokens consumed = learning proxy
- **至少兩個模型**：避免 overfitting 到單一 AI 的怪癖
  - 作者經驗：從 Claude 3.5 Sonnet 擴展到 20+ 個模型
  - 預算有限可用 OSS model 作為 backup

### 3.2 找到組織中的關鍵人物

- **Mavens**：喝第一杯咖啡前就已經 burn 百萬 token 的人
- **Connectors**：跨團隊散播技巧的 Staff Engineers
- **Salespeople**：能把週末 hack 變成全場爆滿 demo 的人
- 歷史規律：推動 cloud、CI/CD、DevOps 的通常也是同一批人

### 3.3 建立實驗文化

- 創建 AI-specific channels 讓大家分享經驗
- 舉辦 office hours 和 expert talks
- 低摩擦的 AI 實驗預算

### 3.4 安裝安全護欄

- 不要一次對所有人 roll out AI
- 先讓 tipping-point contributors 產生成功案例
- 對所有 AI-generated code 要求額外的 validation
- 「Jessie 值班時不准 vibe coding」原則
- 明確的 ownership 標準 + escalation path

### 3.5 說一個英雄故事

- 找一個 bounded, high-value 的 backlog item
- 讓 pilot team 用 vibe coding 攻下它
- 當他們用 1/10 時間交付時，把 demo 投到大螢幕上
- 案例：線上博彩公司用 Python（非團隊主力語言）建 ID verification prototype → 直接上 production

### 3.6 常態化 Blameless Post-Mortems

- 每次事故都會被怪罪到「AI」——順勢而為
- 公開 retrospective、記錄教訓、建立防護措施
- 避免「個人把 AI 省下的時間據為己有」的經濟學陷阱

### 3.7 案例研究：Sourcegraph Leaderboard

- Quinn Slack 的 real-time token burn dashboard
- Token usage = AI engagement proxy（類比工廠電力消耗量 = 產出指標）
- VP of Finance Dan Adler 某週有最多 lines of code → 工程師被 peer pressure 推動
- 非技術人員（sales、marketing、CS）也在用 coding agent

### 3.8 AI 時代的招聘

- **首要檢查**：候選人是否已經親自實驗過 AI？如果沒有 → 紅旗
- **溝通能力**從 nice-to-have 變成 non-negotiable
- **Code review at scale**：每天讀 10,000+ 行 code 的能力
- **現場 AI 實作評估**：讓候選人用 AI assistant 解題，觀察 prompt framing、context management、debugging
- **資歷可能不再是主要因素**：Kent Beck — "native vibe coders will be much better than us"
- Steve 建議：至少一場 in-person + 一場 air-gapped（無 AI）面試

## 5. 最難理解的三個部分 (3 Hardest Sections)

### 難點 1：Token Burn 作為 KPI — 數字遊戲還是真指標？

- **為什麼難**：直覺上「消耗 token」跟「創造價值」之間的因果關係很弱。工程師可以跟 AI 聊天八小時但什麼都沒產出。而且 leaderboard 有明顯的可被操弄性 (gameable)。這不會變成另一個「lines of code per day」式的偽指標嗎？
- **白話解釋**：Token burn 不是績效指標（performance metric），而是**參與指標（engagement proxy）**。想像你在推廣一門新的程式語言。你不會用「寫了多少行 code」來衡量學習進度，但你會看「IDE 開了多少小時」。如果一個人的 IDE 從來沒打開過，你知道他根本沒開始學。Token burn 就是「AI 版的 IDE 開啟時間」。Quinn Slack 很清楚這個限制——leaderboard 的目的是「啟動對話」而不是「評估績效」。
- **實務應用**：

```python
# Token burn dashboard 的正確心智模型
# 它是溫度計，不是秤

class TokenBurnDashboard:
    """
    PURPOSE: Conversation starter, NOT performance review
    METRIC: Engagement proxy (like electricity consumption = factory activity)
    """

    def analyze_developer(self, dev_id: str, monthly_stats: dict) -> dict:
        tokens = monthly_stats['tokens_burned']
        code_lines = monthly_stats['lines_generated']
        conversations = monthly_stats['conversation_count']

        # 這些都不是績效分數！是對話觸發器
        signals = {
            'engagement_level': self._classify_engagement(tokens),
            'suggested_action': None,
        }

        if signals['engagement_level'] == 'very_low':
            # 不是「績效差」，而是「可能需要幫助」
            signals['suggested_action'] = (
                "Reach out: Are they stuck? Need training? "
                "Prefer hand-coding? All valid — just check in."
            )
        elif signals['engagement_level'] == 'very_high':
            # 不是「績效好」，而是「可能有技巧可以分享」
            signals['suggested_action'] = (
                "Invite to share: What techniques are working? "
                "Can they demo at next team meeting?"
            )

        return signals

    def _classify_engagement(self, tokens: int) -> str:
        # 閾值因組織而異，重點是 outlier 分析
        if tokens < 1000:
            return 'very_low'
        elif tokens > 500000:
            return 'very_high'
        return 'normal'
```

### 難點 2：安全護欄 vs. 創新速度 — 如何不讓 Guardrails 變成官僚

- **為什麼難**：本章同時要求「大膽實驗」和「安裝安全護欄」，這看起來像是矛盾的。工程師文化天然抗拒流程和限制，特別是當 AI 帶來的「自由感」正是 FAAFO 的核心吸引力時。如何在不扼殺創造力的前提下設立足夠的安全邊界？
- **白話解釋**：滑雪場有護欄是為了讓你敢滑得更快、敢挑更陡的坡，而不是為了限制你。如果沒有護欄，大多數人反而會減速，因為怕掉下懸崖。AI 的安全護欄也是同樣邏輯：有了明確的邊界（哪些可以做、哪些不可以做），工程師反而會更大膽地實驗，因為他們知道最壞情況被控制住了。
- **具體策略**：

```
GUARDRAILS THAT ACCELERATE (加速型護欄):

 +------------------------------------------+
 | LAYER          | GUARDRAIL               |
 +------------------------------------------+
 | Pre-commit     | AI-generated code must   |
 |                | pass linting + type check|
 +------------------------------------------+
 | Pre-merge      | Automated test coverage  |
 |                | >= existing baseline     |
 +------------------------------------------+
 | Pre-deploy     | Canary deployment +      |
 |                | automatic rollback       |
 +------------------------------------------+
 | Runtime        | Error rate monitoring +  |
 |                | auto-alerting to owner   |
 +------------------------------------------+

 GUARDRAILS THAT KILL CREATIVITY (扼殺型護欄):
 x "All AI code needs VP approval before merge"
 x "AI can only be used for test generation"
 x "Must submit justification for each AI session"
 x "Mandatory 3-day review period for AI PRs"

 PRINCIPLE: Guardrails should be AUTOMATED and INVISIBLE
 when things are going well, and only SURFACE when something
 is actually wrong. Like a car's lane-departure warning.
```

### 難點 3：招聘標準的根本轉變 — 「不用 AI 的人」是紅旗？

- **為什麼難**：這觸及工程師的身份認同。很多資深工程師對 AI 持保留態度，認為「真正的工程師靠自己的腦子寫 code」。作者建議面試時把「沒使用過 AI」視為紅旗，這對許多人來說太激進了。而且，如何區分「有 legitimate reason 不用 AI」和「缺乏好奇心」？
- **白話解釋**：這不是在說「不用 AI = 差的工程師」。這是在說「不用 AI = 沒有好奇心去嘗試新工具」。就像 2015 年面試一個從沒用過 Git 的人——他們可能是很強的工程師，但你會想知道為什麼。也許他們之前的公司禁止使用，也許他們在一個 Perforce-only 的環境——這些都是合理的理由。但如果理由是「我覺得版本控制沒必要」，那就真的是紅旗了。
- **面試設計建議**：

```typescript
// AI 時代面試的 evaluation rubric

interface VibeCodingInterviewRubric {
  // 第一輪：AI interaction (權重最高)
  aiInteraction: {
    promptQuality: 1 | 2 | 3 | 4 | 5;       // 能否清晰表達意圖
    contextManagement: 1 | 2 | 3 | 4 | 5;    // 能否有效管理 conversation
    debuggingAIMistakes: 1 | 2 | 3 | 4 | 5;  // AI 出錯時的應對能力
    criticalEvaluation: 1 | 2 | 3 | 4 | 5;   // 是否盲目接受 AI 建議
  };

  // 第二輪：Code review at scale
  codeReview: {
    speedReading: 1 | 2 | 3 | 4 | 5;        // 能否快速掃描大量 code
    bugSpotting: 1 | 2 | 3 | 4 | 5;          // 能否發現潛在問題
    architecturalAwareness: 1 | 2 | 3 | 4 | 5; // 能否看到 bigger picture
  };

  // 第三輪：Air-gapped coding (無 AI)
  rawCoding: {
    fundamentals: 1 | 2 | 3 | 4 | 5;         // 基本功是否紮實
    problemSolving: 1 | 2 | 3 | 4 | 5;        // 能否獨立思考
  };

  // 紅旗 / 綠旗
  redFlags: string[];   // e.g., "從沒用過任何 AI tool"
  greenFlags: string[]; // e.g., "能說出 3 個 AI 失敗教訓"
}

// 關鍵洞見：最強的候選人會同時展現這兩件事：
// 1. 能熟練使用 AI 產出大量高品質 code
// 2. 在沒有 AI 時仍然能夠獨立解題
// 缺少任何一邊都是問題
```

## 6. 本章小結與架構啟示

**最重要的一句話**：組織的 AI 轉型不是技術問題，是文化問題。而文化從領導者「自己先下廚」開始。

**架構啟示**：

1. **「十小時法則」**。在你對團隊做任何 AI 相關決策之前，先親自用 AI 寫 10 小時的 code。沒有什麼比親身體驗更能建立真正的判斷力。Gene Kim 明確說：10 小時的親手實驗 > 100 頁分析師報告。

2. **Token burn 是採用率的 leading indicator**。就像 CD pipeline 的 deployment frequency 一樣，token burn 告訴你「人們是否真的在用」。但要小心不要把它變成 Goodhart's Law 的犧牲品——一旦成為目標，它就不再是好的衡量標準。所以只用它做 conversation starter，不做 performance review。

3. **Multi-model strategy 是必須的**。一個 AI 模型就像一個 IDE——你可能有偏好，但你應該至少精通兩個。不同模型有不同的 strengths（Claude 寫作正式、Grok/DeepSeek 文風犀利、GPT-4.5/Gemini 寫作優秀、O1/O3 擅長提煉事實）。在你的組織中，至少提供兩個選擇。

4. **Blameless post-mortems 的文化價值被低估了**。當你鼓勵人分享 AI 失敗的故事（「不要像 Steve 那樣弄丟整個 test suite」），你就在建立一個安全學習的環境。反之，如果人們因為 AI 事故而被懲罰，他們就會隱藏 AI 使用——然後你就會面對最糟糕的情況：人們偷偷用 AI 但不承認。

5. **面試流程必須進化**。讓候選人在面試中使用 AI 不是作弊，而是「literally how they'll do their job」。同時，保留至少一場 air-gapped 面試來確認候選人的基本功。2025 年以後，「完全不會在沒有 AI 的情況下寫 code」會是一個巨大的紅旗——同樣地，「完全不會使用 AI」也會是。

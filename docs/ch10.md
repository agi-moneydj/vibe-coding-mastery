## 1. 概述 (Overview)

本章聚焦 vibe coding 最關鍵的技術能力：**context engineering**。作者用「剪貼簿 / 砧板」的比喻，讓讀者理解 context window 的本質——它不是無限的記憶體，而是一個會滿、會退化、使用成本呈二次方增長的有限資源。核心警告：**context 越多，AI 越笨。** 本章提供了 focused vs. comprehensive 兩種 context 策略，以及何時使用 RAG、summarization documents 等進階手段。

對有經驗的 RD 來說，這是最直接影響日常工作品質的一章——你每天和 AI 互動時如何管理 context，決定了 AI 回應的品質上限。

## 2. 學習目標 (Learning Objectives)

完成本章後，你應該能夠：

- **辨識** context saturation 的早期警訊（AI 開始忘記指令、自相矛盾、繞圈子）
- **理解** context window 的二次方成本模型，以及 input/output window 的不對稱性
- **實作** focused context 和 comprehensive context 兩種策略的切換判斷
- **應用** ruthless context culling、checkpoint、compact 等技巧來延長有效對話壽命

## 3. 章節大綱 (Outline)

### 3.1 Context Window 基礎
- Token 的定義：約 4 個字元（「四字詞」助記法）
- Window 大小光譜：512 tokens（BERT）→ 200K-1M tokens（frontier models）→ 10M（Llama 4）
- 80% 的 Git repositories 能放進 500K token window
- 工具：GitIngest / Repomix / files-to-prompt

### 3.2 Context Window 的結構
- System prompt（模型提供者放入）
- 你的 rules & initial prompt
- 程式碼與專案 context
- 對話歷史（線性增長 → 累計 token 成本二次方增長）

### 3.3 Context Saturation 的危險
- 研究顯示：大多數模型在 3,000 tokens 就開始退化
- 退化症狀：忘記細節、自相矛盾、忽視指令
- Steve 團隊的案例：明確禁止 `ps aux`，context 到 50% 時 AI 仍然執行

### 3.4 Output Context Window 限制
- 大多數模型：4,000-8,000 tokens output
- Gemini 2.5 Pro 例外：64,000 tokens
- 實務影響：不能一次生成整個 codebase

### 3.5 兩種 Context 策略
- **Focused Context**：最小必要 context，適合 leaf node 任務
- **Comprehensive Context**：全量 context，適合架構決策 / 小專案
- 實務中通常從 focused 開始

### 3.6 大型 Codebase 的 Context 策略
- Summarization documents（CliffsNotes for your project）
- RAG（retrieval-augmented generation）
- Agentic search（Claude Code 團隊發現有時比 RAG 更好）

### 框架二：Token 成本的二次方增長

```
  累計 Token 消耗
    ^
    |                                        /
    |                                      /
    |                                    /
    |                                  /    <-- 二次方增長
    |                               /         (每次都重傳所有歷史)
    |                            /
    |                         /
    |                     /
    |                 /
    |            /
    |        /
    |     /
    |  /
    +----------------------------------------> 對話 Turn 數

    Turn 1: 送 1 份歷史
    Turn 2: 送 2 份歷史
    Turn 3: 送 3 份歷史
    ...
    Turn N: 送 N 份歷史

    累計 = 1 + 2 + 3 + ... + N = N*(N+1)/2 ≈ O(N^2)

    比喻：油漆工每次要把之前的籬笆重漆一遍
```

## 5. 最難理解的三個部分 (3 Hardest Sections)

### 難點 1：為什麼 Context Saturation 在 3,000 Tokens 就開始？

- **為什麼難**：模型廣告宣稱 200K-1M token window，但研究顯示 3,000 tokens 就開始退化。這個落差讓人困惑——window size 的意義是什麼？
- **白話解釋**：想像你有一張巨大的書桌（200K tokens），理論上可以攤開一百本書。但你的「注意力」只能同時處理眼前的幾頁。書桌上堆越多東西，你找到需要的那頁就越難。Context window 是書桌大小，attention quality 才是你的閱讀能力。
- **實務影響**：

```python
# 錯誤心態：
# "模型有 200K window，我還有很多空間"

# 正確心態：
# "每多一份 context，AI 的推理品質就下降一點"

# 實驗驗證（作者建議的 Alice 實驗）：
conversation = [
    {"role": "user", "content": "Alice is a doctor in Boston who drives a blue Lexus."},
    # ... 插入多篇 Wikipedia 文章 ...
    {"role": "user", "content": "What does Alice do for a living?"},
    # AI 可能回答："I believe Alice is a lawyer" 或 "I'm not sure"
]

# 對策：
# 1. 關鍵指令放在 context 的開頭 AND 結尾（sandwich pattern）
# 2. 定期開新對話
# 3. 使用 compact/checkpoint 功能
```

### 難點 2：Input 和 Output Window 的不對稱性

- **為什麼難**：大家關注 input context window，卻忽略 output 只有 4K-8K tokens。這個不對稱性直接限制了 AI 單次能產出的程式碼量。
- **白話解釋**：你的砧板很大（input），但你的盤子很小（output）。就算你準備了一整桌食材，一次只能上一小盤菜。想做滿漢全席？你必須一道一道上。
- **程式碼示範**：

```typescript
// 這就是為什麼你不能說 "幫我寫一個完整的 microservice"
// 並期待一次拿到全部程式碼

// 正確做法：分步生成
// Step 1: "生成 data model 和 database schema"
// Step 2: "基於上面的 schema，生成 repository layer"
// Step 3: "生成 service layer，使用 step 2 的 repository"
// Step 4: "生成 controller 和 route handlers"
// Step 5: "生成 integration tests"

// 每一步都在 output window 限制內
// 且每一步都可以獨立 review 和測試

// 提示：Gemini 2.5 Pro 的 64K output 是例外
// 如果需要大量一次性生成，考慮用 Gemini
```

### 難點 3：RAG vs. Agentic Search — 哪個更好？

- **為什麼難**：作者自己都說「我們還不知道哪個方法最好」。Claude Code 團隊發現 RAG 在某些情況下反而降低了 coding performance。這對已經投資 RAG 基礎建設的團隊是個意外。
- **白話解釋**：RAG 像是圖書館的索引系統——你搜什麼，它就找什麼。但如果索引建得不好，或者你的問題不適合用關鍵字搜尋，它反而會找到錯誤的資料塞進 context，造成「噪音汙染」。Agentic search 則像一個聰明的研究助理，他會自己翻閱檔案、跑 grep、看目錄結構，雖然慢一點，但可能找到更相關的東西。
- **架構師建議**：

```python
# 目前的實務建議（2025 年中）：

# 1. 小型 codebase (< 500K tokens)
#    → Comprehensive context（全部丟進去）
#    → 不需要 RAG

# 2. 中型 codebase (500K - 5M tokens)
#    → Summarization documents + Focused context
#    → 讓 AI 生成各模組的摘要文件
#    → 按需載入相關摘要

# 3. 大型 codebase (> 5M tokens)
#    → 嘗試 RAG 和 Agentic search 兩者
#    → 衡量哪個在你的場景效果更好
#    → 透過 MCP 接入 RAG 系統

# 產生 summarization document 的 prompt：
SUMMARIZE_PROMPT = """
分析以下模組的程式碼，產生一份結構化摘要，包含：
1. 模組職責（一句話）
2. 公開 API（函數簽名 + 用途）
3. 關鍵設計決策
4. 與其他模組的依賴關係
5. 常見的使用模式

程式碼：
{module_code}
"""
```

### 五個立即可執行的 Context 管理實踐

1. **開新對話的閾值要低**：感覺 AI 開始「飄」了？不要試圖挽救，開新對話重新開始。沉沒成本謬誤在這裡是最大的敵人。

2. **善用 checkpoint / compact**：現代 coding agent（Claude Code、Cursor 等）都支援 checkpoint。每完成一個 milestone 就 checkpoint，走錯路時可以回退。

3. **Sandwich 你的關鍵指令**：把最重要的約束條件放在 context 的開頭（system prompt / CLAUDE.md）**和**每次 prompt 的結尾。AI 對開頭和結尾的注意力最強。

4. **建立 project summarization documents**：

```typescript
// 專案結構摘要範例（存為 ARCHITECTURE.md 供 AI 參考）
/*
## Project: IceDataServicesAPI
## Tech: .NET 8 / Dapper / SQL Server

### 模組結構：
- Controllers/ → API endpoints (bond price, risk-free rate, etc.)
- Services/    → External ICE Data API integration
- Repositories/ → Database access via Dapper + stored procedures
- Middleware/  → IP whitelist security

### 關鍵約束：
- 使用 SemaphoreSlim 控制併發（API rate limiting）
- Connection strings 加密存儲
- 所有 DB 操作通過 stored procedures
*/
```

5. **監控你的 token 使用**：如果你的工具顯示 token count，注意它何時超過 context window 的 50%。那是開始降級的時間點。

### 架構師的 Meta-Insight

Context management 的挑戰本質上是一個 **資訊架構問題**。你在設計 AI 互動時，面臨的取捨和設計 API、設計資料庫 schema 時一樣：

- 資訊太少 → AI 缺乏必要 context，產出品質差
- 資訊太多 → AI 被噪音淹沒，推理品質下降
- 資訊結構不良 → AI 找不到需要的資訊，即使它就在 context 裡

這和系統設計中的 "right-sizing" 原則一模一樣。能在 context management 上做出正確判斷的 RD，通常也是架構設計能力強的人——因為底層的思維模型是相通的。

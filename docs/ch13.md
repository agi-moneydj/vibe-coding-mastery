## 1. 概述 (Overview)

開發工具的格局正在經歷史無前例的劇變。Steve 用了 Emacs 三十年，然後在一週內幾乎完全切換到 AI coding agent，六個月後 Emacs 又回來了——但角色完全不同，變成了 agent orchestration hub。本章幫助你在這片混亂中建立清晰的導航地圖：什麼時候用 coding agent、什麼時候用 chat assistant、IDE 在哪些場景仍然不可取代，以及 MCP（Model Context Protocol）如何成為連接 AI 與你所有工具的「新 HTTP」。

核心訊息：**你不是你的編輯器**。你的真正資產是多年累積的經驗和直覺，而不是任何一個特定工具。

## 2. 學習目標 (Learning Objectives)

完成本章後，你應該能夠：

- **辨識** 不同 AI 工具模態（agent、chat assistant、IDE、chatbot）各自的最佳使用場景
- **理解** MCP 的 client/server 架構，以及它為何被稱為「新 HTTP」
- **實作** 一個基本的 MCP server，讓你的 AI agent 能夠存取自訂工具和資料來源
- **建立** 面對工具快速迭代的健康心態，避免「工具認同危機」
- **評估** 何時該嘗試新工具、何時該堅持現有工具的決策框架

## 3. 章節大綱 (Outline)

### 3.1 開發工具的寒武紀大爆發
- **舊時代**：選 IDE 像買房子，一住數十年（IntelliJ、VS Code、Emacs）
- **新時代**：工具生命週期從「數十年」壓縮到「數月」
- **Steve 的工具旅程**：Emacs 30 年 → IntelliJ 分走 50% → AI agent 一週內取代 → Emacs 回歸（agent orchestration）
- **加速的變化**：OpenAI Codex + o4-mini 發布後，40 分鐘內就開始考慮放棄用了 7 週的 Claude Sonnet 3.7

### 3.2 IDE、Agent 還是 Terminal？
- **Coding Agent 適合的場景**
  - 大多數非平凡任務
  - 自主迭代直到完成
  - 比 chat 的「來回貼文字」模式更高效
- **Chat Assistant 適合的場景**
  - 快速診斷（Gene 的 2 分鐘 bug fix 故事）
  - 即時 boilerplate 生成
  - 已有完整 context 的簡單問題
- **IDE 仍然勝出的場景**
  - 大型 codebase 的語義索引和導航（LSP）
  - 確定性 refactoring（rename、extract method）
  - 超大型 codebase（百萬行以上）
  - IntelliJ MCP server：讓 agent 存取 IDE 的 refactoring 工具
- **Chatbot 仍然勝出的場景**
  - 深夜 brainstorming（Gene 遛狗時用 iPad ChatGPT）
  - 概念性問答和規劃
  - 24/7 可用的顧問

### 3.3 在工具動盪中生存的原則
- **你不是你的編輯器**：真正的資產是經驗和直覺
- **和同行交流**：比較筆記，分享什麼工具有效
- **無法全部嘗試**：有策略地試用有前景的新工具

### 3.4 Model Context Protocol (MCP)：連接 AI 與你的工具
- **MCP 是什麼**：AI agent 存取外部工具和服務的標準協定
- **MCP in Action**：Steve 用 MCP/Puppeteer 讓 AI 操作遊戲 UI
- **為什麼 MCP 重要**：從「知識豐富的顧問」升級為「能動手的團隊成員」

### 3.5 MCP 技術實作
- **架構**：Client（AI assistant）→ Server（adapter）→ Service（你的工具）
- **協定細節**：JSON-RPC 2.0 over HTTP/WebSocket
- **建立 MCP Server**：list_tools()、type annotations、tool definitions
- **快速上手**：ModelContextProtocol.io 的 weather server 範例

## 5. 最難理解的三個部分 (3 Hardest Sections)

### 難點 1：MCP 到底是什麼，為什麼它這麼重要

- **為什麼難**：MCP 聽起來像又一個 API 標準，工程師們已經見過太多「改變一切」的協定最後不了了之。更混亂的是，MCP 的概念同時涉及 client/server 架構、JSON-RPC、tool discovery、type annotation 自動生成等多個層面，很容易迷失在技術細節中而忽略了它的戰略意義。

- **白話解釋**：想像你有一個超強的新實習生（AI），但他被關在一個房間裡，只能看你給他的文件。MCP 就是在牆上開了一扇窗和一扇門——窗讓他看到外面的世界（databases、APIs、running applications），門讓他走出去操作（click buttons、fill forms、run queries）。微軟 CTO Kevin Scott 說「MCP 是 HTTP 的道德等價物」，意思是：就像 HTTP 讓所有電腦能夠溝通一樣，MCP 讓所有 AI 能夠操作任何工具。

- **實作範例**：一個最小的 MCP server

```python
# my_first_mcp_server.py
# 使用 FastMCP 建立一個讓 AI 查詢你的內部 API 的 MCP server

from fastmcp import FastMCP

mcp = FastMCP("my-internal-tools")

@mcp.tool()
def get_deployment_status(service_name: str, environment: str) -> dict:
    """
    查詢內部服務的部署狀態。

    Args:
        service_name: 服務名稱，例如 "user-service", "payment-api"
        environment: 部署環境，例如 "staging", "production"

    Returns:
        包含版本號、部署時間、健康狀態的 dict
    """
    # MCP 框架會自動從 type annotations 和 docstring
    # 生成 tool definition，AI 就知道怎麼呼叫這個 tool
    return {
        "service": service_name,
        "env": environment,
        "version": "2.3.1",
        "deployed_at": "2025-01-15T10:30:00Z",
        "healthy": True,
        "replicas": 3,
    }

@mcp.tool()
def run_database_query(query: str, database: str = "readonly_replica") -> dict:
    """
    在唯讀 replica 上執行 SQL 查詢。
    注意：僅限 SELECT 語句，不允許 mutation。

    Args:
        query: SQL SELECT 語句
        database: 目標資料庫 (預設: readonly_replica)
    """
    # 安全檢查
    if not query.strip().upper().startswith("SELECT"):
        return {"error": "只允許 SELECT 查詢"}

    # 實際執行查詢的邏輯...
    return {"rows": [], "row_count": 0, "execution_time_ms": 42}

# 啟動 server
if __name__ == "__main__":
    mcp.run()

# 在你的 AI client 設定中加入：
# {
#   "mcpServers": {
#     "my-internal-tools": {
#       "command": "python",
#       "args": ["my_first_mcp_server.py"]
#     }
#   }
# }
```

### 難點 2：IDE 還有沒有未來？什麼時候該用什麼工具？

- **為什麼難**：這是一個帶有情感色彩的問題。許多工程師花了數年甚至數十年精通一個 IDE，它的 keybindings 已經刻入肌肉記憶。書中說「你不是你的編輯器」，但要放下這個認同並不容易。更困難的是，答案不是「全面切換」而是「情境選擇」——每種工具都有它仍然勝出的場景。

- **白話解釋**：想像你是一個專業廚師，你有一把用了 20 年的主廚刀。現在來了一台食物處理機（AI agent），它能比你快 10 倍切菜。你會完全扔掉主廚刀嗎？不會。精細的裝飾切工（code navigation in huge codebase）你還是用刀。大量粗切（feature development）用食物處理機。偶爾你還是想用手揉麵團（手動 debug 來學習）。重點不是「哪個工具最好」，而是「哪個工具最適合這個任務」。

- **決策 checklist**：

```typescript
// tool-selection-guide.ts

type TaskCharacteristic = {
  codebaseSize: "small" | "medium" | "large" | "google-scale";
  taskType:
    | "new-feature"
    | "bug-fix"
    | "refactor"
    | "navigation"
    | "planning"
    | "testing";
  urgency: "minutes" | "hours" | "days";
  contextNeeded: "full-project" | "single-file" | "conceptual";
};

function recommendTool(task: TaskCharacteristic): string {
  // Agent：自主開發，迭代完成
  if (
    task.taskType === "new-feature" ||
    task.taskType === "testing"
  ) {
    return "Coding Agent (Claude Code, Cursor Agent, Amp)";
  }

  // Chat：快速診斷，2 分鐘解決
  if (
    task.taskType === "bug-fix" &&
    task.urgency === "minutes"
  ) {
    return "Chat Assistant (IDE inline chat)";
  }

  // IDE：確定性操作，大規模 codebase
  if (
    task.taskType === "refactor" ||
    task.taskType === "navigation" ||
    task.codebaseSize === "large" ||
    task.codebaseSize === "google-scale"
  ) {
    return "IDE (IntelliJ / VS Code) — 確定性 refactoring + LSP";
  }

  // Chatbot：概念性、規劃性任務
  if (
    task.taskType === "planning" ||
    task.contextNeeded === "conceptual"
  ) {
    return "Chatbot (ChatGPT / Claude.ai) — 24/7 brainstorming";
  }

  return "Coding Agent (default for most tasks)";
}
```

### 難點 3：Steve 的工具變遷故事到底想告訴我們什麼

- **為什麼難**：表面上看，Steve 的工具歷程只是個人軼事——用了 30 年 Emacs，換了，又換回來。但深層訊息是關於「身份認同的流動性」和「工具選擇的策略思維」。很多工程師讀完會感到焦慮：「如果工具變化這麼快，我投資在學習任何工具上還有意義嗎？」

- **白話解釋**：Steve 的故事是一個「螺旋式回歸」的案例。Emacs 第一次離開時，它是「打字用的編輯器」。回來時，它變成了「agent orchestration 的中樞」。工具的形式沒變（還是 Emacs），但用途完全不同了。這就像電話的演化——1990 年你用它打電話，2025 年你用它做幾乎所有事情，但形式上它還是叫「phone」。教訓是：**不要投資在工具的 surface-level mastery 上，要投資在理解工具解決什麼問題的 meta-skill 上。**

- **實用原則**：

```
  工具投資的 ROI 模型：

  高 ROI（長期不變的能力）          低 ROI（可能很快過時）
  +-------------------------------+-------------------------------+
  | - 架構設計能力                  | - 特定 IDE keybinding         |
  | - Code review 的眼力            | - 特定 agent 的 prompt 技巧    |
  | - Testing 策略思維               | - 特定工具的 config 格式       |
  | - 問題拆解能力（task graph）     | - 特定 model 的行為怪癖        |
  | - 快速學習新工具的能力           |                               |
  | - MCP/protocol 設計理解          |                               |
  +-------------------------------+-------------------------------+

  核心原則：
  1. 投資在 "meta-skills"（學習如何學習工具）
  2. 和同行交流（compare notes frequently）
  3. 保持多個逃生通道（master multiple approaches）
  4. 讓 AI 幫你學新工具（它已經知道怎麼用了）
```

## 6. 本章小結與架構啟示

### 對資深工程師的核心啟示

1. **MCP 是你現在就該投資的技術**：不管你用什麼 AI tool，MCP 都會是連接它們到你的基礎設施的標準方式。微軟 CTO 把它比作 HTTP，這不是誇張。學會寫 MCP server 就像 2000 年代初學會寫 REST API 一樣，是一個高杠杆的技能投資。

2. **IDE 不會死，但角色會轉變**：IDE 的語義索引（LSP）能力在 AI 時代更重要了，因為 AI 也需要這些索引來理解你的 codebase。IntelliJ 已經提供 MCP server，這意味著 AI agent 可以使用 IDE 的確定性 refactoring 工具。未來的 IDE 可能更像是「AI 的工具箱」而非「人的工作台」。

3. **Gene 的 2 分鐘 bug fix 故事是最佳實踐範本**：會議前 15 分鐘發現 bug → IDE 複現 → chat assistant 貼 stack trace → AI 指出問題 → 推 production → 2 分鐘完成。這不是「AI 取代人」，而是「人 + AI 的最佳協作模式」——人提供 context（stack trace、知道問題在哪），AI 提供 solution。

4. **「你不是你的編輯器」是最重要的心態**：工具會來來去去，但你的架構思維、debugging 直覺、系統設計經驗是永恆的。不要因為工具快速變化而焦慮，要把這種變化視為機會——你的經驗讓你比新手更快評估和採用新工具。

5. **Escape hatch 原則**：永遠保持多種方法的熟練度。當你的主要工具遇到瓶頸時（就像 Steve 在 Gradle 上遇到的），你需要能夠「gracefully degrade」到其他工具。這不是對新工具的不信任，而是專業工程師的風險管理。

### 一句話總結

> 工具的寒武紀大爆發不是威脅，而是機遇——前提是你把自己定位為「工具的指揮家」而非「某個工具的使用者」。MCP 是你的通用接口，經驗是你的護城河。
